# auto v0.3.0 — Outcomes-Gated Emission

## Problem Frame

v0.2.0 shipped **recipes** — named workflow topologies declared as JSON.
Two of the built-in recipes (A2 Parallel Theories + Judge, A4 Adversarial
Pair + Comparator) are **structurally round-gated**, not outcomes-gated:

- **A2** declares `units: [plan-1, plan-2, plan-3, judge]`. If the judge
  finds all 3 theories weak, the engine cannot spin up 2 more theories and
  re-judge. The "3" is baked into the recipe JSON at init time.
- **A4** declares `units: [plan, build-clarity, build-perf, compare]`. If
  the comparator can't decide between clarity and perf, the engine cannot
  add a third bias and re-compare. The "2 builders" is structural.

This cuts against auto's identity. Auto's whole value proposition is
**durability + outcomes-gating** — the disk-persisted ledger drives forward
motion, the exit predicate decides when to stop, and the loop pattern
"work until the predicate is met" is the canonical shape. Today that
pattern exists in the **work-loop** (verdict → fix → re-review → … until
only minors remain) but is absent at the **plan/judge/comparator layer**.

The v0.2.0 round-2 code review already flagged this gap. From
[[feedback_ce_plan_round2_exposed_producer_gap]]:

> "Dynamic emission has no producer. A3 [Build-First Feedback] drives 40%
> of engine surface."

A3 (Build-First Feedback) was deferred from v0.2.0 because its
`phase_order: ["work_sketch", "review", "plan", "work_refine"]` topology
required the same dynamic-emission machinery this brainstorm targets.
v0.3.0 lands the primitive and A3.

## The Reframe (the load-bearing insight)

The right primitive is NOT "three flavors of dynamic emission" (one per
recipe). It is a **single generalized verb**: *iterate-until-predicate*
as a recipe-declared loop, applied to the plan/judge/comparator layer the
same way auto already applies it in the work-loop's closure pattern.

Auto already iterates until-predicate in the work-loop. v0.3.0 lifts that
pattern UP a level so plan candidates, bias dimensions, and sketch→review
pairs can iterate the same way. The engine's existing closure machinery
(atomic `record_verdict`, `recompute_predicate`, partial-completion-safe
`converge`, harness re-invocation on verdict landing) ALREADY EXISTS. No
new emitter trigger is needed at the architecture level; the existing
verdict-write IS the trigger.

## Actors

- **A1.** The operator running `/auto plan.md --recipe a2` (or a4 / a3).
  Doesn't write recipes; picks them.
- **A2.** The recipe author. Writes JSON (or uses `/auto-author-recipe`)
  declaring an iteration shape: which unit is the gate, what gets
  re-emitted on iterate, what the bound is.
- **A3.** The auto engine. Reads the gate unit's verdict, decides whether
  to advance / re-emit / exit based on the recipe-declared loop.

## Key Flows

- **F1. Outcomes-gated A2 run.** Operator types `/auto plan.md --recipe
  a2`. Engine spawns 3 plan candidates + 1 judge. Judge reviews them and
  emits verdict `{decision: "iterate", emit_count: 2, reason: "all weak"}`.
  Engine re-emits 2 more plan candidates (plan-4, plan-5) with the same
  shape as plan-1/2/3 (the recipe's plan-candidate template). Judge runs
  again on the new set. Repeats up to `max_attempts`. On `decision: "advance"`
  the judge sets `winner_unit_id` and the engine emits work units. On
  `decision: "exit"` (or bound exceeded) the engine exits with the
  best-so-far winner if any, otherwise a no-winner report.

- **F2. Outcomes-gated A4 run.** Operator types `/auto plan.md --recipe
  a4`. Engine spawns 1 plan + 2 builders (clarity, perf) + 1 comparator.
  Comparator emits `{decision: "iterate", emit_bias: "safety"}` or
  similar. Engine spawns a third builder with the new bias and re-runs
  the comparator on all three. Repeats up to `max_attempts`. On
  `decision: "advance"` the comparator names a winner (or a merged
  output). On exhaustion the engine surfaces the comparator's last
  best-so-far.

- **F3. A3 (Build-First Feedback) end-to-end.** Operator types `/auto
  plan.md --recipe a3`. Engine spawns a work_sketch unit; on completion
  spawns a reviewer; reviewer's verdict either says "sketch is good, plan
  the refinement" (advances to plan phase) OR "sketch needs another pass"
  (re-emits a work_sketch unit). Plan phase produces refinement units;
  work_refine phase builds them; standard work-loop closure applies.

## Requirements

### R1 — The `verdict.decision` enum

Gate units (judge in A2; comparator in A4; reviewer in A3) write a verdict
that includes a `decision` field with one of three values:

- **`advance`** — the gate is satisfied; advance to the next phase (or to
  done). Optional payload: `winner_unit_id` (A2), `winner_bias` or merge
  result (A4), refinement_plan (A3).
- **`iterate`** — re-emit per the recipe's iteration shape; repeat the
  gate. Optional payload: `emit_count`, `emit_kind` (for A4's
  bias-naming), `reason` (free-text, surfaced in the report).
- **`exit`** — give up gracefully; the gate has determined no further
  iteration will help. The engine exits with the best-so-far state.

The decision field is the engine's single source of truth for what to do
next. No predicate-string parsing. No DSL.

### R2 — The recipe's iteration declaration

A recipe declares iteration with a new top-level field:

```json
{
  "iteration": {
    "gate_unit": "judge",
    "emit_template": "plan-candidate",
    "bound": {
      "max_attempts": 5,
      "max_wall_seconds": 1800
    }
  }
}
```

- **`gate_unit`** — the unit id whose verdict.decision drives the loop.
- **`emit_template`** — the template id for re-emitted units when the
  gate says iterate. The recipe declares emit templates (e.g.,
  `plan-candidate` shapes the new units the same way plan-1/2/3 were
  shaped at init).
- **`bound.max_attempts`** — integer; the maximum number of iterate
  decisions the engine will honor before forcing exit. Required.
- **`bound.max_wall_seconds`** — optional integer; the maximum ACTIVE
  wall time the engine will honor before forcing exit. Pause-aware (R5).

### R3 — Emit templates

A recipe declares emit templates parallel to its `units` block:

```json
{
  "units": [...],
  "emit_templates": {
    "plan-candidate": {
      "phase": "plan",
      "invokes": { "adapter_op": "next_plan_step" },
      "id_prefix": "plan-"
    }
  }
}
```

When the engine re-emits via the iteration loop, it spawns N units shaped
like the template, with auto-assigned IDs (`plan-4`, `plan-5`, …). The
template is the v0.3.0 producer (closes the "dynamic emission has no
producer" gap from round-2).

### R4 — The bound enforcement

After every gate-unit verdict-write, the engine checks the bound BEFORE
honoring `decision: "iterate"`:

- If `attempts_made >= max_attempts`, force `decision = "exit"` (override
  the unit's verdict). Surface the override in the run report.
- If `max_wall_seconds` is set and `active_wall_seconds >= max_wall_seconds`,
  force `decision = "exit"` likewise.

Bounds are engine-enforced, not honor-system on the unit-agent. This
satisfies [[feedback_deterministic_over_probabilistic_v1]].

### R5 — Active wall-time accounting

`max_wall_seconds` counts **active** time only — wall-clock time during
which the engine is making progress. A seam pause, an operator suspend,
or a long stretch between `/auto-resume continue` invocations does NOT
burn the budget.

The ledger gains two new top-level fields:
- `active_wall_seconds` — accumulated active time since run start
- `last_active_at` — ISO-8601 timestamp of the most recent active-state
  transition

Active state begins when `loop.driver == "self"` and a tick is dispatched
or in flight. Active state ends when the run reaches a seam pause,
loop_phase becomes `done`, or no work is in flight AND no tick is armed.
On resume, `active_wall_seconds` continues counting from where it left
off; `last_active_at` is updated.

### R6 — Composition with the existing exit predicate

The existing `exit_predicate_result.met` continues to gate run-level
exit. The new iteration loop is local to the gate unit's phase. Two
clean interaction rules:

- The iteration loop completing (gate says `advance` or `exit`) does NOT
  mean the run exits — it means the phase advances. The standard
  predicate then gates the run.
- The standard predicate becoming `met` (work-loop's closure) does NOT
  retroactively abort an iteration in progress — once a gate unit
  verdicts, the engine honors its decision.

No conflict possible: the iteration loop and the run-exit predicate
operate on different units and at different times.

### R7 — Backward compatibility

a1 (Plan → Review → Work → Review — the renamed v0.2.0 "Classic CE
Stack") and W (Work-only) do NOT declare `iteration` or `emit_templates`.
Their behavior is unchanged. The engine treats absence of the iteration
declaration as "no outcomes-gated loop on this recipe" — standard phase
flow applies.

A recipe that declares `iteration` but no matching gate_unit, or an
unknown emit_template, fails validation at recipe-load time.

### R8 — A3 finally lands

A3 (Build-First Feedback) ships in v0.3.0 with `phase_order:
["work_sketch", "review", "plan", "work_refine"]`. Its iteration loop is
declared on the reviewer unit: `decision: "iterate"` re-emits a
`work_sketch`; `decision: "advance"` moves the run from review → plan.
A3's specific iteration shape, bound defaults, and emit_templates are
captured in `recipes/a3.json` after this brainstorm progresses to plan.

### R9 — Operator-facing diagnostics

The standard tick INTENT's `operator_guidance` field (shipped in v0.2.0
fix-pass H) gains iteration-aware messages:

- When `attempts_made >= max_attempts - 1` (one attempt left), the
  guidance surfaces a "last iteration before bound" warning.
- When the engine overrides `iterate` due to bound, the guidance names
  WHICH bound triggered (max_attempts vs max_wall_seconds) and the
  best-so-far state.

### R10 — Test coverage

Every iteration-aware recipe ships with:
- A GREEN integration test driving the production write path
  (record_verdict + decision="advance") through to a clean exit.
- An ITERATE integration test (decision="iterate") that verifies the
  engine re-emits the correct unit count and shape.
- A BOUND integration test (max_attempts hit) that verifies the engine
  overrides the unit's `iterate` decision with `exit`.
- A DELIBERATE-FAIL test per [[feedback_new_tests_need_deliberate_fail_smoke_check]]
  — proves the test actually catches the regression class (e.g., assert
  emit_count > 0 when iterating, or that bound enforcement fires).

## Scope Boundaries

### In scope for v0.3.0

- The engine-level primitive (`verdict.decision` + bound enforcement +
  emit templates + active-wall-time accounting)
- A2 converted to use iteration (re-emits plan candidates on weak judge)
- A4 converted to use iteration (re-emits bias-differentiated builders
  on undecided comparator)
- A3 finally ships (build-first feedback with reviewer-driven iteration)
- a1 description renamed from "Classic CE Stack" to "Plan → Review →
  Work → Review" (see [[idea_auto_a1_rename_to_workflow_identity]])
- Documentation updates: `docs/contracts/recipe-format.md`,
  `skills/auto/SKILL.md`, README

### Deferred for later

- **CLI-level predicate override** (`/auto plan.md --until '<predicate>'`).
  The decision lives in the recipe; deferring this honors the "no
  predicate-string DSL" decision and keeps the operator surface clean.
- **Operator-promotable bound overrides** (`/auto plan.md --recipe a2
  --max-attempts 10`). Possible v0.3.1 add if real usage shows recipe-
  declared bounds are too rigid. v0.3.0 keeps bounds in the recipe.
- **Cost-based bounds** (`max_tokens`, `max_cost_usd`). Out of scope —
  the engine doesn't see Anthropic token counts today. Could be a
  v0.4.0 add if usage data justifies it.
- **Cross-iteration learning** (the judge sees prior iterations' findings
  when judging a new batch). v0.3.0's gate units see the current
  ledger state; richer context propagation is a separate question.

### Outside the product's identity

- **A "search" mode where the engine generates candidates without a plan
  input.** Auto starts from a plan or spec. Open-ended generative search
  belongs in a different tool.
- **Multi-agent debate-style iteration** (theories adversarially
  critiquing each other across rounds). Auto's loop is single-gate-unit
  per phase. Multi-agent debate is a different orchestration pattern.

## Open Questions

- **OQ1: emit_template's exact JSON shape.** R3 sketches it; the planner
  should choose between the named-template approach (recipe declares
  templates by id, units reference them) vs an inline-template approach
  (the iteration field carries the template directly). Probably the
  named approach for symmetry with the existing recipe shape.
- **OQ2: best-so-far semantics on bound-exit.** R4 says "force exit,
  surface the override." Does the engine surface the LAST gate verdict's
  payload (e.g., judge's best-so-far winner candidate) or just exit with
  no winner? Probably "surface the last gate's best-so-far if it named
  one, else exit no-winner."
- **OQ3: resume + active-wall-time edge cases.** What happens if the
  ledger has `active_wall_seconds` but the process clock has rolled
  backward (system clock change, NTP correction)? Probably "monotonic
  clock for accumulation, wall clock for reporting only."
- **OQ4: emit_template ID conflicts.** If a recipe declares 3 initial
  plan units and the iteration emits 2 more, the auto-assigned IDs need
  to not collide. Plan: `id_prefix + N` where N starts at the
  count-of-existing-units-matching-the-template + 1.
- **OQ5: A3's exact phase grammar wire.** R8 names A3 as in scope but
  defers the recipe JSON to planning. The planner should specifically
  verify the engine's phase-grammar code path handles
  `["work_sketch", "review", "plan", "work_refine"]` without a
  special-case branch (it should already — fix-pass A.2 generalized
  phase_transitions — but verify before assuming).

## Success Criteria

- **SC1.** An operator running `/auto plan.md --recipe a2` against a
  realistic plan can have the engine iterate plan candidates until the
  judge picks a winner OR the bound triggers. No manual intervention.
- **SC2.** All four iteration-aware recipes (A2, A3, A4 + any
  workspace recipe that opts into iteration) honor the same bound
  semantics. Recipe-declared bounds are mechanically enforced.
- **SC3.** A v0.2.x recipe (a1, W) continues to work unchanged. Suite
  stays green on the unchanged recipes.
- **SC4.** A bound-triggered exit surfaces actionable diagnostics —
  the operator sees which bound triggered, what the best-so-far state
  was, and what to inspect for promotion.
- **SC5.** The build-bug class from v0.2.0
  ([[feedback_plan_documents_transition_code_doesnt_wire_it]]) does not
  recur: every transition this brainstorm documents (verdict-write →
  decision-read → bound-check → re-emit OR advance OR exit) has a
  specific code site identified in planning, and integration tests drive
  the production write path through that site.

## Dependencies & Assumptions

- **Verified:** The v0.2.0 engine's `transition_and_emit` primitive
  (lib/ledger.py) and the recipe-declared phase_transitions mechanism
  (lib/_bootstrap, lib/tick) support arbitrary `to_phase` values today.
  A3's `work_sketch → review → plan → work_refine` phase order should
  work without engine changes beyond the iteration primitive. (To
  re-verify in planning.)
- **Verified:** The v0.2.0 round-3 P3 close (set_winner_unit_id excludes
  judge_unit_id) generalizes to iteration: the new emit_template lookup
  must similarly reject the gate unit naming itself.
- **Assumed:** The harness re-invocation contract from v0.2.0 fix-pass G
  ([[feedback_polling_inside_vs_outside_agentic_loop]]) continues to
  hold — when a gate unit's record_verdict lands, the launching session
  is re-invoked, and that re-invocation is when the iteration check
  fires. v0.3.0 does NOT add polling.
- **Assumed:** Iteration-aware recipes are USED. If A2/A4/A3 in their
  iteration-aware form see no adoption after v0.3.0 ships, v0.3.1 should
  re-examine whether the primitive justifies the surface area.
