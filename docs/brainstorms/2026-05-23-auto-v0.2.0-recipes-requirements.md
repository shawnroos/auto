# auto v0.2.0 — Recipes (packaged workflows for the loop engine)

## Problem Frame

`auto` v0.1.1 ships exactly one workflow shape: plan → seam → work → exit-on-P3. That's the shape the engine encodes today, and it's the only one available to a user typing `/auto <plan>`.

But the user already runs **at least four distinct workflows manually** with Claude primitives + popular workflow plugins (CE, nerd, native /goal):

- **A1** — Plan, then build, then review, then fix to P3-only exit. (What `auto` does today.)
- **A2** — Run N competing plans in parallel from one goal, build them all, judge picks the winner. (`nerd`-style competing theories applied to execution.)
- **A3** — Build a rough sketch first, review *that*, let the findings drive the plan, then refine. (How research code actually evolves.)
- **A4** — Two builders work the same plan with different biases (clarity vs perf, conservative vs ambitious), comparator picks or merges. (How `auto`'s own ce-correctness + ce-adversarial reviewers paired up.)

Each is a real, repeated workflow. Each is currently executed by hand. The cost is that `auto`'s loop engine, ledger, fan-out, and durable resume only serve A1 — the other three get none of that infrastructure.

The frame, in the user's words: ***`auto` is just packaging Claude primitives and popular workflow plugins.*** The unit of packaging is a **recipe**: a named, file-backed declaration of *how* a workflow composes. v0.2.0 ships the recipe format, four built-in recipes encoding the four workflows above, a three-tier registry so recipes can be workspace-local / personal / built-in, and an authoring skill that lets any user describe a workflow in plain language and have an agent compile it into a recipe.

---

## Actors

- A1. **End user (recipe author + runner)**: every `auto` user — not just plugin authors. Picks a recipe at `/auto` start; describes a new workflow in conversation and lets the authoring skill compile it into a recipe; saves recipes personally or per-project.
- A2. **Authoring skill (recipe compiler)**: an agent-invoked skill that elicits a workflow shape from prose, validates against the recipe format, and writes the file. Optionally watches `auto` runs and offers to capture a topology that ran well.
- A3. **`auto` engine**: reads the selected recipe, constructs the initial ledger topology from it, and runs the loop. Recipe-blind once the ledger is built.
- A4. **Picker (selection UI)**: surfaces workspace recipes, then global, then built-ins; renders a short visual preview per recipe; lets the user pick.

---

## Key Flows

- F1. **First-time user picks a built-in recipe and runs `auto`**
  - **Trigger:** User types `/auto <plan>` with no `--recipe` flag in a fresh repo (no workspace recipes yet).
  - **Actors:** A1, A4, A3.
  - **Steps:**
    1. Picker fires AskUserQuestion. Shows the four built-ins (A1/A2/A3/A4) with brief one-line descriptions and an ASCII topology preview per option.
    2. User selects one.
    3. Engine reads the recipe file, constructs the initial ledger from its topology declaration, arms the first tick.
    4. Run proceeds exactly as `auto` v0.1.1 (driven by the constructed ledger).
  - **Outcome:** Run begins with the chosen recipe's topology. Status/resume work identically to v0.1.1.
  - **Covered by:** R1, R2, R5, R8.

- F2. **User authors a new recipe by describing it in conversation**
  - **Trigger:** User invokes the authoring skill (e.g. "I want to save the workflow I just ran" or "I want a new recipe where X").
  - **Actors:** A1, A2.
  - **Steps:**
    1. Skill elicits: goal of the workflow, what runs in parallel vs sequence, how phases are ordered, what severity exit conditions matter, what plugins/commands each unit invokes.
    2. Skill produces a draft recipe file, shows the user the topology in the same ASCII format the picker uses, and asks for confirm/revise.
    3. On confirm, skill asks: save as workspace recipe (this project only) or global (personal, all repos)?
    4. Skill writes the file to the chosen tier.
  - **Outcome:** A new recipe exists, visible in the picker on the next `/auto` invocation.
  - **Covered by:** R3, R4, R6, R7, R10.

- F3. **User runs `/auto` in a repo with workspace recipes already saved**
  - **Trigger:** User types `/auto <plan>` with workspace recipes present in `<repo>/.claude/auto/recipes/`.
  - **Actors:** A1, A4, A3.
  - **Steps:**
    1. Picker resolves the three-tier set (workspace → global → built-in), deduped by name, first-wins.
    2. Picker shows the resolved list with source labels ("workspace" / "global" / "built-in") so the user can tell which they're picking.
    3. User picks; engine runs the chosen recipe.
  - **Outcome:** Workspace recipes are surfaced and shadow same-named global/built-in entries.
  - **Covered by:** R6, R8.

- F4. **Power-user bypass via flag**
  - **Trigger:** User types `/auto <plan> --recipe a2`.
  - **Actors:** A1, A3.
  - **Steps:**
    1. Engine resolves `a2` against the three-tier registry directly, no picker.
    2. Run proceeds.
  - **Outcome:** Picker is skipped entirely. Suitable for scripts and repeat invocations.
  - **Covered by:** R5.

## Requirements

**Recipe format**
- R1. A recipe declares the run's **initial ledger topology**: the set of units, their `depends_on` graph, their phase (`plan` / `work`), and the metadata each unit needs to dispatch (which adapter op or Claude primitive it invokes, severity scale, parallelism cap). The engine reads the recipe and constructs the initial ledger; the engine is otherwise unchanged once the run is underway.
- R2. The four built-in recipes (A1/A2/A3/A4) ship with the v0.2.0 plugin and demonstrate the format's expressive range. Each is a structural test case for the format: A1 is the existing sequential cascade; A2 requires multiple parallel plan-loops + a judge unit; A3 requires phase ordering other than `plan→seam→work`; A4 requires two parallel work-loops + a comparator unit.
- R3. **V1 ships declarative-only.** The recipe format is a single JSON file declaring topology + per-unit metadata; A2's judge and A4's comparator are expressed as ordinary units whose dispatch invokes a specific adapter op with a structured prompt. No `.py` companion file in V1. A `python_hook` field is reserved in the JSON schema for future use, but the V1 engine does not load Python from recipes. Rationale: keeps every user (not just Python-writers) in the author audience, and the "agent writes the file" path stays simple. If A2 or A4 turn out to need genuine free-form logic during implementation, that's a planning-surfaced contract gap to raise — not a V1 deliverable to scope speculatively.
- R4. The format prioritizes **agent-writability over human-writability**. Recipes are written by the authoring skill on the user's behalf; humans read them to debug or hand-tweak but aren't expected to compose them from scratch. The format may be verbose and denormalized in service of validation and engine consumption.

**Registry and resolution**
- R5. Recipes are resolved from a three-tier search path: **workspace** (`<repo>/.claude/auto/recipes/<name>.json`) → **global** (`~/.claude/auto/recipes/<name>.json`) → **built-in** (shipped with the plugin under `recipes/<name>.json`). Deduped by `name`, first-wins. The `--recipe <name>` flag uses the same resolution.
- R6. The picker (and the authoring skill) labels each recipe with its tier so users can tell at a glance which they're picking or shadowing.

**Picker UX**
- R7. Bare `/auto <plan>` — no `--recipe` flag — opens the picker via AskUserQuestion with a side-by-side ASCII preview per option (one or two screens tall, like a DX7 algorithm card). The preview shows the topology in the same format the authoring skill uses when it shows a draft, so users learn one visual vocabulary.
- R8. `--recipe <name>` skips the picker. Default for scripted and repeat invocations.

**Authoring skill**
- R9. The plugin ships a skill (provisional name: `auto-author-recipe`) that elicits a workflow shape from user prose, produces a draft recipe file, shows the topology back as ASCII, and saves to workspace or global on confirm.
- R10. The skill supports two entry points: (a) "I want to encode this workflow from scratch" (skill drives the elicitation from a goal description), and (b) "save the topology of the run that just finished" (skill reads the completed run's ledger and reverse-derives a recipe). Both produce files of the same format.

**Engine extensions to support the format**
- R11. The engine permits **multiple parallel plan-loops** in one run (today: exactly one plan-loop per run). A recipe like A2 declares N parallel plan-loop units; the engine fans them out, gates the work-loop on whatever the recipe declares (a judge unit's verdict, in A2's case).
- R12. The engine permits **non-default phase ordering**. Today the phase grammar is `plan → seam → work`; A3 requires `work (sketch) → review → plan → work (refine)`. The recipe declares its phase order; the engine reads it instead of assuming the default. This is the most invasive engine change in v0.2.0 and is the one most likely to surface contract gaps against the existing ledger schema (R-IDs in `Outstanding Questions`).

**Compatibility and migration**
- R13. Existing v0.1.x behavior is preserved exactly when no recipe is selected and no `--recipe` flag is passed, and the only built-in is A1. *(This is the regression-free guard — if a user upgrades from v0.1.1 to v0.2.0 and never touches the picker, their workflow is byte-for-byte identical because A1 is the encoded form of v0.1.1's behavior.)*
- R14. The existing `--adapter ce|native` flag continues to work and is orthogonal to recipes: a recipe selects topology + which units invoke which ops; the adapter still supplies the operation content. A recipe can pin a specific adapter or leave it open.

## Acceptance Examples

- AE1. **Covers R7, R13.** Given a user on `auto` v0.2.0 with no workspace or global recipes saved, when they type `/auto docs/plans/foo.md`, the picker shows exactly four entries (A1/A2/A3/A4, all labeled "built-in") with one-line descriptions and an ASCII topology preview per option. If the user picks A1, the run is indistinguishable from v0.1.1 against the same plan.

- AE2. **Covers R5, R6.** Given a workspace recipe `<repo>/.claude/auto/recipes/team-review.json` exists with `name: team-review`, when the user invokes the picker, `team-review` appears at the top labeled "workspace." If the user also has `~/.claude/auto/recipes/team-review.json` with different contents, the workspace version wins and the global version is omitted from the list (not shadowed-and-shown, just omitted) with no user-visible warning.

- AE3. **Covers R9, R10.** Given a user types `/auto-author-recipe "two builders, same plan, one optimizes for clarity and one for perf, then a comparator picks"`, the skill produces a recipe file whose topology matches A4's. Concretely: two `work`-phase units with empty `depends_on`, plus a third `work`-phase comparator unit with `depends_on` listing both. The skill shows the ASCII topology and asks the user to confirm before writing.

- AE4. **Covers R11.** Given recipe A2 declared with three parallel plan-loops + one judge unit, when the run starts, the ledger shows three units in `dispatched` (plan-phase) immediately after the first tick. After all three return verdicts, the judge unit dispatches; the work-loop does not begin until the judge unit is `terminal`.

- AE5. **Covers R12, R13.** Given recipe A3 declared with phase order `[work_sketch, review, plan, work_refine]`, when the run starts, the first dispatched unit is the sketch (a work-phase unit), not a plan-phase unit. The exit predicate is evaluated with the same `met` rules as today, but the phase transitions follow the recipe's declared order. A1 still runs in the default `plan → seam → work` order with no recipe-side phase declaration needed.

## Success Criteria

- A user who already runs A2, A3, or A4 by hand reports that running them under `auto` v0.2.0 reproduces their manual workflow and adds the durable-ledger, fan-out, and resume that the manual version lacks.
- The authoring skill produces a working recipe from a one-paragraph prose description of a new workflow, without the user touching JSON. The skill's first successful end-to-end use is the test that the format is at the right level of abstraction.
- v0.1.x users upgrading to v0.2.0 see no behavioral difference when they continue to invoke `/auto <plan>` and pick the default. The picker is *new*, but the default is *the same*.
- A downstream `/ce-plan` agent reading this brainstorm can produce an implementation plan without inventing the recipe format's grouping, the resolution rules, the four built-in recipes' topologies, or whether the engine changes for R11/R12 are in or out of scope.

## Scope Boundaries

- **Mid-run recipe switching is OUT.** A run picks one recipe at start; the topology is fixed for the run's lifetime. Switching mid-run is a state-machine nightmare; defer to a future version if it ever becomes necessary.
- **Recipe inheritance / diffing is OUT.** A recipe is a self-contained file. No `extends: a1` semantics, no diff format. The authoring skill handles "I want A1 but with a tweak" by writing a fresh complete recipe, not a partial.
- **A community / shared recipe registry is OUT.** v0.2.0 supports workspace + global + built-in. Cross-user sharing happens via committing workspace recipes to a repo, or copying files. A central "recipe hub" with discovery, ratings, etc. is a separate product question, not v0.2.0.
- **A GUI / web picker is OUT.** AskUserQuestion in the terminal is the picker. ASCII previews are the visual vocabulary. No Electron, no web app, no separate config UI.
- **Auto-recommending a recipe from the plan's content is OUT.** The picker doesn't try to LLM-suggest "based on your plan, A2 seems right." The user picks. (Deferred — interesting for v0.3.0 once we see what recipes people actually write.)
- **Telemetry on which recipes are used is OUT.** No usage tracking in v0.2.0. The product question of "which recipes get used and why" is something we learn from talking to users, not from anonymous data.

## Key Decisions

- **The audience for authoring recipes is every `auto` user, not just plugin authors.** This decision drove the choice to make the format agent-writable rather than human-writable and to make the authoring skill a V1 deliverable rather than an afterthought. (Established 2026-05-23 in dialogue.)
- **The recipe file is the contract; the JSON format is a compile artifact.** Users describe workflows in prose; agents compile to the file. The JSON optimizes for engine consumption and validation, not for hand-editing. This is an inversion of where I initially put the design pressure.
- **All four built-in recipes ship together in v0.2.0.** Engine extensions for A2 (parallel plan-loops) and A3 (non-default phase ordering) are in-scope. Trimming to A1+A4 was considered and rejected — the user has evidence-backed demand for all four and the picker abstraction is hollow without topology variety to pick between.
- **Three-tier registry: workspace shadows global shadows built-in.** Standard resolution pattern. Recipes can be project-bound or personal; the picker labels each so the user knows what they're picking.
- **Picker UX is AskUserQuestion with ASCII topology previews.** Same visual vocabulary used by the authoring skill when it shows drafts back — learn-one, use-twice.

## Dependencies / Assumptions

- The locked **adapter contract** (`docs/contracts/adapter-contract.md`) is unchanged in v0.2.0. Recipes select which adapter ops each unit invokes, but the op set, signatures, and severity semantics stay the same. *(Verified against the contract file 2026-05-23.)*
- The **ledger schema** (`docs/contracts/ledger-schema.md`) needs minor extensions to support R11 (multiple parallel plan-loops, which today is a singleton concept) and R12 (non-default phase ordering, which today is hard-coded `plan→seam→work`). These extensions are additive — existing ledger files from v0.1.x runs remain readable. *(Outstanding question on whether the schema extension is straightforward or surfaces a contract gap — see "Resolve Before Planning.")*
- The `auto` engine's `unit_is_terminal`, `recompute_predicate`, and orchestrator `dispatch_batch` continue to work without modification — they're already scale-aware and `depends_on`-driven; the new topologies use both more aggressively but don't introduce new mechanics.
- AskUserQuestion supports side-by-side preview content with multi-line ASCII (verified in this session). The picker depends on this affordance; if it changes, the picker falls back to numbered options in chat (already specified by `feedback_slash_command_nl_routing_pattern`).
- The user already runs the four target workflows manually. This is the evidence-gap closure — the demand for A2/A3/A4 is observed, not speculated.

## Outstanding Questions

### Deferred to Planning

- [Affects R1, R11, R12] [Technical] The exact ledger schema extension to support parallel plan-loops and non-default phase ordering. `ledger-schema.md` is the source of truth; the extension must be additive (existing v0.1.x ledgers remain valid). Best resolved in planning by reading the schema and proposing minimal-additive changes.
- [Affects R2] [Technical] The exact topology declaration for each of A1/A2/A3/A4 in the recipe format. Once R3's declarative-vs-Python question is resolved, write each built-in as the format's canonical example — they double as documentation.
- [Affects R9, R10] [Needs research] The authoring skill's elicitation flow — what questions it asks, in what order, to compile a recipe from prose. This is the V1 UX deliverable but it's a skill-design question that benefits from sketching against the four built-ins as test cases first.
- [Affects R12] [Technical, possible contract gap] Whether non-default phase ordering (A3) interacts cleanly with the existing exit predicate. Today the predicate is phase-aware (`loop_phase == "work"` requires `all_units_terminal AND ...`; `loop_phase == "plan"` requires `gaps_open==0 AND plan_step=="review_plan"`). A3 has TWO work phases (sketch + refine) and a plan-phase between them. The predicate may need a new clause, or the recipe may need to declare which phase is "terminal."
- [Affects R11] [Technical] The picker's behavior when no recipes are resolvable (e.g. `--recipe nonexistent`, or no built-ins shipped). Probably "error with the resolved set so the user can re-try" — but worth a concrete spec.
