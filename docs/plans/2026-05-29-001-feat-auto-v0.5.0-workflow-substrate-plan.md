# auto v0.4.x escape-hatch — observability-led recovery surface

## Anchor evidence — the anticipation argument (review r5 finding 11)

The substrate mechanism (recipe declares off-script policy in prose
→ compiled to try/catch in JS) handles only deviations the recipe
author ANTICIPATED. The motivating incident was, by Shawn's own
framing, a deviation the recipe did NOT anticipate. **Pre-enumeration
cannot catch what was not foreseen.**

This is the load-bearing reason the inversion is correct. It is
ALSO the load-bearing reason the escape-hatch must NOT lead with
pre-enumerated typed states: the same critique applies — any
specific list of "new states we should add" is a guess at what the
next unanticipated shape will look like, and guesses about the
unanticipated have a poor base rate.

The honest mechanism for unanticipated incidents is a generic
stuck-detection + recovery surface — operator-visible state +
free-text guidance + a no-typed-state-required `/auto-resume` path.
NOT a richer enumeration on either side (more recipe try/catch OR
more ledger states). New types are justified ONLY when ≥2
production incidents of the same shape have been observed AND
generic recovery is demonstrably worse for that shape.

## Competing diagnoses (review r5 finding 12)

Four candidate diagnoses fit Shawn's symptom "a unit failed in a
way the recipe didn't anticipate and I had to manually unstick":

1. **State model** — engine vocabulary too narrow. The framing
   both the substrate and the early escape-hatch drafts took.
2. **Recovery ergonomics** — recoverable states are reachable but
   their RECOVERY SURFACE is hostile (`/auto-resume` requires more
   context than the operator has at the moment of intervention;
   ledger edits are needed because the recovery path doesn't
   expose the right verbs).
3. **Observability** — the state IS represented, but the operator
   cannot SEE what state the run is in or why it's stuck, so
   recoverable situations LOOK like dead ends.
4. **Adapter layer** — the failure was in `/ce-work` or `/ce-plan`
   returning an output auto couldn't classify, and the fix lives
   in the adapter contract, not in auto.py at all.

At 2-ledger evidence the diagnosis is unresolvable. **Lean toward
(2) + (3): they help regardless of which diagnosis is dominant,
and they AVOID committing engine-shape changes ((1)) until
evidence is in.** (4) deserves a one-pass audit of adapter return
contracts as part of the catalogue step; if the incident
originated downstream of auto, this whole plan is the wrong layer.

## v0.4.x escape-hatch — Plan of Record

**Goal:** close the rigidity-anecdote pain by shipping
**generic operator-visible recovery first**, and adding typed
states ONLY where ≥2 observed incidents of the same shape justify
typing. The fix to "auto feels rigid" is operator surface + free-
text guidance, not engine-shape changes on a 2-ledger evidence base.

### Scope (concrete)

**1. Generic recovery surface — LEAD deliverable (review r5
finding 5).** Ship these regardless of catalogue outcome:

- `/auto-status` surfaces a "stuck" view: any unit whose state has
  not changed for >N minutes is rendered with — last unit info, its
  raw output snippet, the explicit next-action lines (`/auto-resume
  retry-stuck <run>`, `/auto-abort <run>`), and any free-text
  operator message written by on-stop. This works for ALL stuck
  shapes including ones we have not categorised.
- `/auto-resume retry-stuck <run>` — new subcommand. Enumerates
  units in actionable-stuck states (initially: `stalled` past
  threshold; `dispatched` past stall-threshold without re-arm;
  `verdict-returned` with non-fixable findings) and re-dispatches
  them per the appropriate `ALLOWED_TRANSITIONS` edge. No
  per-unit-id needed — the operator does not need to read the
  ledger to recover. (Review r5 finding 6.)
- `on-stop.py` is and remains READ-ONLY (lock-free atomic-rename
  snapshot per its docstring; review r5 finding 2). The free-text
  operator-facing message is written by the appropriate mutator
  inside `_with_locked_ledger` (see step 3 for the writer site
  map), not by on-stop. on-stop's role is to RENDER the message
  it finds on next session start, not to author it.
- `loop.last_progress_at` field added — tracks when any unit last
  changed state (vs. the existing `last_beat_at`, which tracks
  driver liveness). Stuck detection in `/auto-status` keys on
  `now - last_progress_at > N`. (Review r5 finding 7
  observability-delta spec.)

**2. Catalogue observed off-script shapes (evidence-gating step).**

Walk `<repo_root>/.claude/auto/<run-id>.json` ledgers (currently 2
distinct in `brand-foundry-backend/.claude/auto/`); enumerate
side-quest narratives from session histories. Catalogue against
the REAL closed vocabulary from `lib/ledger_core.py:107–114`:

> `pending, dispatched, verdict-returned, fixed, stalled,
> terminal-skip`

(Review r5 finding 1 — the earlier draft cited a phantom
`failed-with-verdict` state.) Each observed shape gets: (a)
frequency, (b) which existing state surfaced it (likely `stalled`
from dispatch-timeout, `verdict-returned` with non-fixable
findings, or a unit stuck in `dispatched` past its stall threshold
without any tick re-arm), (c) WHY the generic recovery surface
from step 1 was insufficient for this specific shape.

**3. Add typed states ONLY if evidence-gated (review r5 finding
5).** A typed state is justified iff ≥2 incidents of the same
shape are observed AND step 1's generic recovery is demonstrably
inadequate for that shape. If the catalogue yields zero
justified states, this plan ships as observability-only (step 1
+ status renderer changes only).

For each justified state, the touch list is (review r5 finding 3
— it is NOT a single-file mutator add):

- `lib/ledger_core.py` — `UNIT_STATES` (line 107–114),
  `ALLOWED_TRANSITIONS` (line 119–126), `unit_is_terminal`
  (line 330), `_normalize_unit` (line 962–964), and
  `_evaluate_met` / `recompute_predicate` (lines 433, 535) so the
  new state does NOT silently flip `exit_predicate.met` to true.
- `lib/ledger_mutators.py` — a new transition mutator (e.g.
  `mark_malformed`), and `_VERDICT_WRITABLE_STATES` if applicable.
  Per review r5 finding 6, add the retry edge into
  `ALLOWED_TRANSITIONS` (e.g. `malformed-verdict → pending`,
  `timeout-no-verdict → pending`, `spawn-failed-retryable →
  dispatched`) so `/auto-resume retry-stuck` can route it.
- The WRITER site (review r5 finding 2 — on-stop CANNOT write):
  - dispatch-timeout / per-unit-stall classification → `lib/tick.py`
    (stall_threshold path).
  - verdict shape validation → adapter scripts
    (`lib/adapter-ce.py`, `lib/adapter-native.py`) and/or
    `lib/ledger_mutators.record_verdict`.
  - cmux-spawn failure → `lib/orchestrator.py` (dispatch_batch
    path).
- `lib/auto-status.py` (review r5 finding 8 — Python, not the .md):
  render new states with explicit `next: /auto-resume retry-stuck`
  guidance lines (lines 317–320, 341–344).
- `docs/contracts/ledger-schema.md` — §3 (state grammar) and §5
  (I-2 predicate semantics).
- `tests/unit/ledger-mutators.test.sh` — exercise new edges.

**4. Substrate-independent prep work — DEFERRED (review r5
finding 17).** The ~50-LOC ledger library-wrapper noted in the
parked RFC is NOT in scope here. The escape-hatch is small enough
that the existing 4-file ledger split is not the bottleneck;
extracting a public-API wrapper now would be premature abstraction
at the live LOC budget. If the substrate RFC is ever re-activated,
the wrapper lands there.

### Invariants the escape-hatch MUST preserve

- **I-1 (lost-update prevention, review r5 finding 15):** any new
  retry path that READS ledger state then WRITES must do so inside
  a single `_with_locked_ledger` invocation. See
  `lib/ledger_core.py:718–721` for the lock contract; see parked
  RFC 002 (KTD-3 cross-runtime section) for the lost-update
  analysis that applies equally to in-process retry routes.
- **on-stop READ-ONLY contract:** on-stop never opens a writer.
  Per its docstring (`lib/on-stop.py:5–6`), it operates lock-free
  via atomic-rename snapshot. Any message it surfaces was written
  by a mutator under lock.
- **Predicate semantics (I-2):** any new state added must update
  `unit_is_terminal` so it does NOT masquerade as terminal and
  silently flip `exit_predicate.met` to true.

### Risks (review r5 finding 14)

- **R1 state grammar breaks I-2 silently.** New state forgotten
  in `unit_is_terminal` → `all_units_terminal` flips true with
  units still actionable → false "done." MITIGATION: any state
  add includes a `unit_is_terminal` update in the same commit;
  test fixture exercises a per-state terminal-classification
  assertion.
- **R2 retry races with active tick chain.** `/auto-resume
  retry-stuck` flips state to `pending`; a tick chain still
  running may also try to re-arm the same unit. MITIGATION: the
  retry path runs inside `_with_locked_ledger`; the tick chain
  already serialises through the same lock. Verify with a
  synthetic concurrent-retry test fixture.
- **R3 observability delta too small to feel different.** If
  `last_progress_at` + stuck view ship but the operator still
  doesn't recover faster, the plan failed its goal. MITIGATION:
  measure mean-time-to-recovery on the next 5 manual-unstick
  incidents post-ship; if it has not improved, revisit the
  diagnosis (Competing Diagnoses section).
- **R4 evidence base too thin.** 2 ledgers is sub-statistical.
  MITIGATION: catalogue is gated; observability ships
  unconditionally; new states are evidence-gated; the failure mode
  is "we shipped only observability and learned more from
  production usage," which is acceptable.

### Test Strategy (review r5 finding 14)

- Unit tests for any new `ALLOWED_TRANSITIONS` edges
  (`tests/unit/ledger-mutators.test.sh`).
- Smoke test that new states (if any) render in `/auto-status`
  with the correct `next:` guidance line.
- Round-trip test for `/auto-resume retry-stuck` against a
  synthetic stuck-ledger fixture: produce a ledger with units in
  each actionable-stuck state; run `retry-stuck`; assert all
  units transitioned per the edge map.
- Concurrent-retry test (R2 mitigation): two `/auto-resume`
  invocations against the same ledger serialise correctly via
  `_with_locked_ledger`.
- on-stop READ-ONLY regression: assert on-stop never holds a
  writer flock (mock-based test).

### Sequencing

1. Catalogue (step 2) → drives whether step 3 fires at all.
2. Observability (step 1) ships unconditionally and FIRST: it is
   the LEAD deliverable per finding 5; it pays for itself even at
   zero typed states added.
3. State grammar additions (step 3, conditional) — only if
   catalogue evidence justifies.
4. Writer sites in `tick.py` / `orchestrator.py` / `adapter-*.py`
   wired AFTER the grammar lands.
5. `/auto-resume retry-stuck` lands after grammar + writers, so
   it has both the edge to traverse and the state to find units
   in.
6. `/auto-status` rendering update lands LAST so the operator
   surface reflects the fully-wired pipeline.
7. Tests authored alongside each step; full suite gates the PR.

### Success Criteria

- `/auto-status` surfaces a stuck view with explicit next-action
  lines for every actionable-stuck unit, regardless of whether
  new typed states were added.
- `loop.last_progress_at` is written on every state change and
  rendered in `/auto-status`.
- `/auto-resume retry-stuck <run>` round-trips on the synthetic
  fixture: enumerates stuck units, re-dispatches them per the
  correct edge, leaves the ledger in a re-runnable state.
- on-stop remains lock-free read-only (regression test passes).
- IF typed states were added: each new state has a corresponding
  `unit_is_terminal` classification, a `_normalize_unit` entry,
  predicate-semantics handling, a writer site, a retry edge, and
  a render line — all asserted by tests.
- Mean-time-to-recovery on the next observed manual-unstick
  incident is measured and recorded (success: any reduction;
  failure to improve triggers a Competing-Diagnoses revisit).
- `bash tests/run.sh all` passes green.
