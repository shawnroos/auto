# feat: auto v0.2.0 — Recipes (packaged workflows for the loop engine)

## Overview

`auto` v0.2.0 introduces **recipes** — named, file-backed JSON declarations of workflow topology. A recipe describes the initial ledger the engine builds (which units exist, their `depends_on` graph, which phase each runs in, and what each invokes). The engine reads the recipe and builds the ledger; everything downstream of that (tick loop, dispatch, predicate evaluation, resume) runs unchanged for A1 and extended for non-default topologies.

v0.2.0 ships:

1. The **JSON recipe format** + a single validator used by both the engine loader and the authoring skill.
2. **Four built-in recipes** (A1/A2/A3/A4) encoding workflows the user already runs manually.
3. A **three-tier registry** (workspace → global → built-in) with first-wins resolution.
4. A **picker** at `/auto` start via `AskUserQuestion`.
5. An **authoring skill** (`auto-author-recipe`) that compiles recipes from prose.
6. **Two engine extensions** documented as additive contract changes: R11 (multiple parallel plan-loops) and R12 (non-default phase ordering).

Behavioral guarantee: v0.1.x users see no behavioral change. A1 is the canonical encoding of v0.1.1's behavior — picking A1 in the picker (or accepting the default) produces a first-tick ledger byte-identical to v0.1.1's.

---

## Problem Frame

v0.1.1 ships exactly one workflow shape — plan → seam → work → exit-on-P3 — and the user already runs at least four distinct workflows manually using Claude primitives + popular plugins. v0.2.0 packages those workflows as recipes so `auto`'s durable ledger, fan-out, and resume can serve all four. The frame, from the brainstorm: ***`auto` is just packaging Claude primitives and popular workflow plugins.*** (see origin: `docs/brainstorms/2026-05-23-auto-v0.2.0-recipes-requirements.md`)

## Requirements Trace

- R1. Recipes declare initial ledger topology — units + `depends_on` + phase + per-unit metadata.
- R2. The four built-in recipes ship in v0.2.0 and demonstrate the format's expressive range.
- R3. V1 ships declarative-only JSON. `python_hook` is a reserved schema field that the V1 engine does not load.
- R4. Format prioritizes agent-writability over human-writability.
- R5. Three-tier search path: workspace → global → built-in; deduped by `name`, first-wins.
- R6. Picker labels each recipe with its tier so users see what they're picking.
- R7. Bare `/auto <plan>` opens the picker via `AskUserQuestion`.
- R8. `--recipe <name>` skips the picker.
- R9. Plugin ships `auto-author-recipe` skill that compiles recipes from prose.
- R10. Skill supports both prose-elicitation and reverse-derive-from-completed-run entry points.
- R11. Engine permits N parallel plan-loops in one run.
- R12. Engine permits non-default phase ordering.
- R13. v0.1.x behavior preserved exactly when picking A1 (or default).
- R14. `--adapter ce|native` continues to work; orthogonal to `--recipe`.

**Origin actors:** A1 (end-user / recipe author + runner), A2 (authoring skill), A3 (auto engine), A4 (picker).
**Origin flows:** F1 (first-time built-in pick), F2 (author from prose), F3 (workspace shadows global), F4 (`--recipe` flag bypass).
**Origin acceptance examples:** AE1 (default picker behavior; covers R7/R13), AE2 (workspace shadows global; covers R5/R6), AE3 (skill produces A4-shape from prose; covers R9/R10), AE4 (A2's parallel plan-loop ledger shape; covers R11), AE5 (A3's non-default phase ordering; covers R12/R13).

## Context & Research

### Relevant Code and Patterns

- **The single change point for recipe→ledger:** `lib/auto.py:194` — `run()` calls `ledger.init_ledger(repo_root, run_id, adapter=adapter, units=[], loop_phase="plan")`. Today `units=[]` and `loop_phase="plan"` are hardcoded. Recipes replace those two argument values. `init_ledger` (`lib/ledger.py:489`) already accepts arbitrary `units` and `loop_phase`; the caller just never supplies them.
- **Phase grammar enforcement (the R12 surface):** Three load-bearing sites all key on the literal `"plan"` vs `"work"` distinction:
  - `lib/tick.py:533-562` — `_tick_body` 3-way phase switch
  - `lib/tick.py:634-671` — `_maybe_seam` hardcodes plan→seam(→work) transition
  - `lib/ledger.py:315-352` — `recompute_predicate` binary plan-vs-everything-else branch
  - Plus secondary touchpoints at `lib/tick.py:508` (predicate-met preempt) and `lib/tick.py:610` (post-advance recheck).
- **Plan-loop singularity (the R11 surface):** Enforced by three top-level scalars + one adapter call: `ledger.plan_step` and `ledger.exit_predicate_result.gaps_open` (scalars, one per run); `lib/tick.py:376 advance_plan_loop` calls `adapter.next_plan_step(ledger_dict)` which returns ONE step; today `units=[]` in plan phase so dependency machinery is dormant.
- **Engine seam (unchanged in v0.2.0):** `lib/orchestrator.py::dispatch_batch`, `ready_units`, `_is_ready`, `_dependency_satisfied`, `_ancestor_ids`, `converge`; `lib/ledger.py::unit_is_terminal`, `record_verdict`, `transition`, `_atomic_write`, `_with_locked_ledger`; `lib/tick.py::advance_work_loop`, `_ready_fix_unit`, `_ready_reenqueue_unit`, `detect_and_halt_stalled`. These read only `units[].state`/`depends_on`/`findings[]`/`adapter_scale` and stay topology-agnostic.
- **Commands→lib dispatch pattern:** Three commands today (`auto.md/auto-status.md/auto-resume.md`) each with NL routing prose in the .md and one canonical `bash "${CLAUDE_PLUGIN_ROOT}/lib/X.sh" "$ARGUMENTS"` dispatch line. The smoke test (`tests/smoke/scaffold.test.sh:87-109`) enforces this exactly-once invariant. The picker logic for R7 lives entirely in `commands/auto.md`'s NL routing prose — no new dispatch lines, no smoke-test changes.
- **Test harness:** `tests/run.sh` collects `tests/{smoke,unit,integration}/*.test.sh`. Drop a new `tests/unit/recipes.test.sh` and it auto-runs. Helpers at `tests/helpers/test-helpers.sh`. Pattern examples: `tests/unit/ledger.test.sh` (875 lines), `tests/unit/orchestrator.test.sh` (640 lines).
- **Plugin manifest auto-globs:** `.claude-plugin/plugin.json` declares `skills: "./skills"` and `commands: "./commands"`. Adding `skills/auto-author-recipe/SKILL.md` requires no manifest edit.
- **Contracts (locked, additive-only extension convention):** `docs/contracts/ledger-schema.md` already exercises the additive-extension pattern (the `attempt` field from Bug #6 — schema §2.3 documents `attempt: 0` default and notes "additive / backward-compatible — an old ledger with no `attempt` field reads as `0`"). v0.2.0 uses the same shape for all new fields.

### Institutional Learnings

- **`feedback_plan_documents_transition_code_doesnt_wire_it`** is the anchor learning for this plan. R11 and R12 are exactly the "prose-says-X, engine-must-now-honor-X" shape that bit v0.1.1 six different ways. Every recipe topology rule must be wired through a single centralized helper, not pattern-matched at each consumer. Allocate review-loop budget to post-build code review, not pre-build doc deepening.
- **`feedback_fix_the_class_not_the_cited_instance`**: grep the v0.1.1 engine for every site that branches on `loop_phase` (predicate, `_tick_body`, `_maybe_seam`, status reporting, resume). All sites route through the new phase-grammar helper. Round-1 fixing only cited sites is a known failure mode.
- **`reference_claude_loop_task_daemon_primitives`**: each background agent self-writes its verdict to the ledger atomically; A2's judge unit reads verdicts from disk, not from in-memory `backgroundTasks`. The same pattern applies to A4's comparator.
- **`feedback_slash_command_nl_routing_pattern`** + **`feedback_slash_command_arg_substitution`**: picker logic and `--recipe` flag parsing live in `commands/auto.md`'s NL routing prose; `lib/auto.sh` stays headless. The .md body keeps the exactly-one-`$ARGUMENTS`-line invariant.
- **`feedback_loop_monitor_terminal_state_field`**: A3 has two work-phases. Without a `terminal_phase` field, the predicate fires after the sketch work-phase and the run silently exits. Required schema field for any non-default `phase_order`.
- **`feedback_predicted_bugs_need_tests_not_conventions`** + **`feedback_deterministic_over_probabilistic_v1`**: validation is mechanical, not conventional. JSON Schema validator gates the recipe at load AND at skill-write time. **Accept the reserved `python_hook` field but ignore it; reject ALL OTHER unknown top-level fields.** (Per R3, `python_hook` is a future-compat reservation, not an error.) The "reject unknown fields" principle holds for everything outside the explicit reserved list.
- **`feedback_new_tests_need_deliberate_fail_smoke_check`**: every new test (AE1–AE5 + schema validator tests + non-default phase-order tests) gets a deliberate-fail check via the **Edit tool**, not a sed/regex revert. Particularly load-bearing for AE5 (sketch-fires-first) which is structurally easy to silently green-pass.
- **`feedback_subagent_write_verification`**: the authoring skill, after writing the recipe file, reads it back and runs the validator before reporting "saved." Same for any subagent-dispatched implementation units in this plan.
- **`feedback_review_loop_catches_narrowing`**: budget ≥3 code-review rounds. Claude-modes V2 took 9; claude-dispatch v0.1.1 took 3+.

### External References

None. Pure Python 3 stdlib + bash + on-disk JSON; no framework dependencies. The locked contracts in `docs/contracts/` are the canonical reference.

## Key Technical Decisions

- **KTD-1. A1 is the canonical encoding of v0.1.1, not a fallback. Phase 1 removes the legacy hardcoded `init_ledger` call.** Old code path is removed when the recipe pipeline lands. R13's regression test — "first-tick ledger from A1 recipe ≡ first-tick ledger from v0.1.1 invocation (modulo additive fields)" — doubles as the recipe-pipeline integration test. **Trade explicitly named:** the alternative was one release of transient overlap (v0.2.0 ships both paths; v0.2.1 removes legacy) so the two paths organically diff-test each other across every v0.1.x run a user makes. Rejected because (a) two implementations of A1 silently drift, (b) the R13 regression test is strong if expanded to "every v0.1.x integration test still passes when re-run through the recipe path" — which is the de-facto coverage. Cost being paid: the R13 regression test's coverage must be exhaustive on day one; no organic diff-test safety net during the transition.
- **KTD-2. Base validator + optional lint pass.** The JSON Schema validator lives in `lib/recipes.py` as `validate(recipe_dict)`. The authoring skill calls a wrapping `validate_and_lint(recipe_dict)` that adds editorial checks the engine ignores: "phase declared in `phase_order` but no unit assigned to it," "depends_on creates an unreachable subgraph," "terminal_phase has no units assigned." Both layers live in `lib/recipes.py`; the contract is `validate`, the editorial pass is `lint`. **This sharpens the original "one validator, two callers" decision** — consistency at the contract layer is preserved (skill output that passes `validate` is engine-acceptable); skill UX gets quality-error catching at write time without the engine over-rejecting recipes that happen to have unreachable units (which is a future-compat hedge, not a correctness violation).
- **KTD-3. Phase-grammar helper enforcement: accessor function + AST lint, NOT one-shot grep at review.** Every site that today branches on `loop_phase` reads from `lib/phase-grammar.py::current_phase(ledger)` instead of `ledger["loop_phase"]`. The helper is the SOLE consumer of the raw field. **Mechanical enforcement** (not convention): (a) a unit test that AST-parses every `lib/*.py` module and asserts `ledger["loop_phase"]` and `ledger.get("loop_phase")` patterns appear ONLY in `lib/phase-grammar.py`; (b) a deliberate-fail check that adds a literal `loop_phase` read to a different module, runs the AST lint, confirms RED. Convention + grep at code review is the **rejected weaker option** — memory `feedback_deterministic_over_probabilistic_v1` says load-bearing pieces need mechanical enforcement on the V1, not disposition shifts.
- **KTD-4. Plan-loop units in N>1 mode execute SERIALIZED. Adapter contract unchanged in V1.** (Resolves G4.) The orchestrator advances exactly one plan-loop unit per tick; other ready plan units stay `dispatched` but idle. The top-level scalar `plan_step` mirrors the actively-advancing unit's value during its tick. **AE4's "3 units in `dispatched` immediately after the first tick"** stands — it means dispatched-from-orchestrator-perspective; the adapter sees one at a time. **Concurrent advance is deferred to v0.3.0** when the adapter contract gets a planned re-lock to add a `unit_id` or `unit_view` parameter to `next_plan_step`. V1's serialized advance is correct and simple; concurrent is an optimization with a contract cost. **The "scalar mirror" framing is dropped** — `plan_step` is the active-advancing-unit's value during its tick, not a "mirror" of anything; the scalar IS the field, and the orchestrator owns which unit is active.
- **KTD-5. Phase-transition emitters are an explicit named-function registry in `lib/emitters.py`; recipes don't declare emitter names in V1.** (Resolves G1.) The engine maps `(from_phase, to_phase)` tuples to emitter function names. V1 ships exactly 4 emitters covering the four built-ins. The implicit-vs-explicit fork: **rejected the "recipes declare emitters" option** because it requires authoring-skill users to understand emitter shapes; the engine ships the small set; v0.3.0 can add a recipe-declared emitter field if the implicit set becomes too narrow. **The decision visible**: A1/A2/A3/A4's "// emitted dynamically" comments now correspond to entries in the HLTD's phase-transition emitter table.
- **KTD-6. Two predicate fields: `met` (run-met) and `phase_met` (phase-advance).** (Resolves G2.) `met` retains today's meaning; new `phase_met` field drives phase advancement; both recomputed in the same atomic write. The original framing "predicate reads terminal_phase: `met` requires `current_phase == terminal_phase AND <rest>`" was incomplete — it didn't separate the two semantics. With two fields, the run-exit and phase-advance paths read distinct fields and neither can silently fire the other.
- **KTD-7. Per-unit `plan_step` and `gaps_open` for plan-phase units (R11).** A1's single plan-loop continues to use the top-level scalars for byte-identical compatibility (no per-unit fields needed on a 1-plan-unit ledger). A recipe with N plan-loops uses per-unit fields. The `recompute_predicate` plan branch reads per-unit when present, scalar otherwise. **Combined with KTD-4's serialized advance, the per-unit fields are written one at a time (the actively-advancing unit's plan_step gets updated each tick).**
- **KTD-8. `terminal_phase` is required for any recipe with non-default `phase_order`.** Validator enforces this. Predicate's `met` field reads it: `met` requires `current_phase == terminal_phase AND <existing scale-aware rules>` (plus the new phase-met fix from KTD-6).
- **KTD-9. Picker default: short-label options with one-line descriptions.** Multi-line ASCII previews via `AskUserQuestion`'s preview field land as the picker's full form **if** verified to render. Otherwise, the picker uses short labels and includes a hint "type `/auto --recipe a1` to skip the picker." The visual vocabulary doc lives in the repo; users can read it once and learn it; the picker doesn't have to render it inline if the surface doesn't support multi-line.
- **KTD-10. Single ASCII topology renderer at `lib/topology-render.py`, called by picker + authoring skill + auto-status.** (Pattern-recognition finding: 3 surfaces drawing ASCII topology cards is the same "two-validators" anti-pattern the plan explicitly guards against.) The renderer takes `(recipe_dict, width_hint) → str`. `skills/auto-author-recipe/references/visual-vocabulary.md` is the renderer's contract doc, not a separate spec.
- **KTD-11. `recipe` and `phase_order` are top-level ledger fields, not per-recipe metadata.** Once a run starts, the ledger carries the recipe's name + phase grammar so status/resume can render them without re-loading the recipe file. Schema extension is additive: missing `recipe` → recipe-blind v0.1.x ledger (treated as A1); missing `phase_order` → default `["plan", "seam", "work"]`.
- **KTD-12. Resume re-reads the ledger, not the recipe file.** `/auto-resume` already reads the ledger fresh per `feedback_slash_command_nl_routing_pattern`. The ledger's `recipe` + `phase_order` fields carry forward all topology info; the recipe file is only read at `init_ledger` time. **Resume across recipe file deletion** is therefore safe — the recipe is "baked in" to the ledger at run start.

## Open Questions

### Resolved During Planning

- **Q1 — Ledger schema extension shape**: Additive per-unit fields (`units[].plan_step`, `units[].gaps_open`) for plan-phase units when N>1; top-level `recipe`, `phase_order`, `terminal_phase` fields. All default such that v0.1.x ledgers are A1-equivalent.
- **Q2 — Topology declarations for A1/A2/A3/A4**: See Implementation Unit U7 for concrete JSON.
- **Q3 — Adapter contract change for R11**: Not needed in V1. **Plan-loop units in N>1 mode execute SERIALIZED** (per KTD-4 — the previous "scalar mirror" framing was incoherent under truly concurrent advance and has been replaced). The top-level `plan_step` field reflects the actively-advancing unit's value during its tick; other plan units stay `dispatched` but idle. Concurrent advance becomes possible in v0.3.0 with an adapter-contract re-lock adding a `unit_id` parameter to `next_plan_step`.
- **Q4 — Phase-aware exit predicate for A3**: Add required `terminal_phase` to recipe schema; predicate reads it.
- **Q5 — Picker on no-resolvable recipes**: `--recipe nonexistent` → exit 1 with resolved set printed. No built-ins case (user deleted them) → exit 1 with hint to reinstall.
- **Q6 — Dynamic unit emission (G1)**: Explicit named-function registry in `lib/emitters.py`. V1 ships 4 emitters: `plan_output_to_work_units`, `judge_winner_to_work_units`, `review_findings_to_plan_input`, `plan_output_to_paired_builders`. The engine maps `(from_phase, to_phase)` tuples to emitter names. Recipes don't declare emitter names in V1 (deferred to v0.3.0 if implicit set proves too narrow). See HLTD "Dynamic unit emission" section for the full table.
- **Q7 — Predicate semantic conflation (G2)**: Two distinct fields in `exit_predicate_result`: `met` (run-met, terminal-phase-only) and `phase_met` (current-phase-met, drives advancement). Recomputed in same atomic write. Old ledgers read `phase_met` as `null`; engine falls back to met-equals-phase-met for recipe-blind A1.
- **Q8 — Atomic snapshot ordering (G3)**: Inside one ledger atomic write: (1) apply triggering transition, (2) run phase-transition emitter, (3) recompute predicate on post-emission unit set, (4) atomic write. New ledger I-1 corollary documented in U11's contract update.

### Deferred to Implementation

- **The exact AskUserQuestion preview surface for ASCII multi-line content.** First implementation pass tries the preview field with multi-line ASCII; if rendering breaks, falls back to short labels per KTD-9. The fallback is the default behavior; the multi-line preview is the upgrade. Test both; ship whichever renders.
- **Exact prose-elicitation flow for the authoring skill (R9).** Sketched in U9; refined during implementation by running the skill against the four built-in topologies as test cases (AE3 is the conformance test).
- **Whether the judge's verdict-write convention in A2 needs a new adapter op or can reuse `record_verdict`.** Initial pass: judge unit uses `record_verdict` like any other unit; its findings array carries the chosen winner (which the `judge_winner_to_work_units` emitter reads to know which plan's output to emit). If that's too restrictive in practice, surface as v0.3.0 work.
- **The exact `emits` field shape (if any) on recipe-declared units, when v0.3.0 lifts the implicit-emitter restriction.** V1 hardcodes the 4 emitters; v0.3.0 may let recipes declare custom emitters. The contract for that field is deferred until we see a user-authored recipe that needs it.

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

### Recipe JSON shape (canonical)

A1 (Classic CE Stack) as the simplest concrete instance:

```jsonc
{
  "name": "a1",
  "version": "1",
  "description": "Classic CE Stack — plan → build → review → fix to P3-only exit",
  "phase_order": ["plan", "seam", "work"],
  "terminal_phase": "work",
  "default_adapter": "ce",
  "units": [
    {
      "id": "plan",
      "phase": "plan",
      "depends_on": [],
      "invokes": {"adapter_op": "next_plan_step"}
    }
    // work units appear dynamically — they're emitted by the plan-loop into units[]
    // at runtime; the recipe doesn't pre-declare them because plan output produces them
  ]
}
```

A2 (Parallel Theories + Judge):

```jsonc
{
  "name": "a2",
  "phase_order": ["plan", "seam", "work"],
  "terminal_phase": "work",
  "units": [
    {"id": "plan-1", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}},
    {"id": "plan-2", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}},
    {"id": "plan-3", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}},
    {"id": "judge",  "phase": "work", "depends_on": ["plan-1", "plan-2", "plan-3"],
     "invokes": {"adapter_op": "review", "prompt_template": "judge.md"}}
    // work units from chosen plan emitted dynamically after judge resolves
  ]
}
```

A3 (Build-First) — illustrates non-default phase ordering:

```jsonc
{
  "name": "a3",
  "phase_order": ["work_sketch", "review", "plan", "work_refine"],
  "terminal_phase": "work_refine",
  "units": [
    {"id": "sketch",  "phase": "work_sketch", "depends_on": [], "invokes": {"adapter_op": "do_unit", "prompt_template": "sketch.md"}},
    {"id": "review",  "phase": "review", "depends_on": ["sketch"], "invokes": {"adapter_op": "review"}},
    {"id": "plan",    "phase": "plan",   "depends_on": ["review"], "invokes": {"adapter_op": "next_plan_step"}}
    // refine work units emitted dynamically after plan resolves
  ]
}
```

A4 (Adversarial Pair + Comparator):

```jsonc
{
  "name": "a4",
  "phase_order": ["plan", "seam", "work"],
  "terminal_phase": "work",
  "units": [
    {"id": "plan", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}}
```

### Dynamic unit emission (G1 — phase-transition emitters)

The JSON sketches above carry `// emitted dynamically` comments. That dynamic emission is the single most important wiring point in v0.2.0 — if it's hand-waved, A2/A3/A4 ship as JSON that validates but silent-fails at runtime, suite green, recipes broken. This subsection is the explicit specification.

**Decision: emitters are an explicit registry, recipe-blind for V1.** A new module `lib/emitters.py` owns a registry of named emitter functions. The engine maps `(from_phase, to_phase)` tuples to emitter names (the "trigger" column in the table below is documentation of which verdict transition fires that emitter — it's not part of the dispatch-table key). The recipe's phase ordering implies which emitter fires; the recipe does NOT declare emitter names directly (that's deferred to v0.3.0 if the implicit set proves too narrow). V1 ships exactly the emitters the four built-ins need.

Phase-transition emitter table for the four built-ins:

| Recipe | Phase boundary | Trigger | Emitter | New unit shape | Owning function |
|---|---|---|---|---|---|
| A1 | `plan` → `seam` → `work` | plan-loop-met | `plan_output_to_work_units` | adapter's plan output → 1 unit per top-level plan item | `lib/emitters.py::plan_output_to_work_units` |
| A2 | judge unit `terminal-no-blockers` → `work` (chosen) | judge verdict-returned | `judge_winner_to_work_units` | judge's findings carry the winning plan-loop unit id; emit the winner's plan output as work units | `lib/emitters.py::judge_winner_to_work_units` |
| A3 | `review` verdict → `plan` dispatch | review verdict-returned | `review_findings_to_plan_input` | review findings shape next plan-loop's input; emit one new plan-phase unit | `lib/emitters.py::review_findings_to_plan_input` |
| A3 | `plan` met → `work_refine` | plan-loop-met (in middle position) | `plan_output_to_work_units` (reused from A1) | same as A1, units have phase `work_refine` not `work` | `lib/emitters.py::plan_output_to_work_units` |
| A4 | `plan` met → `work` | plan-loop-met | `plan_output_to_paired_builders` | plan output → 2 work units (clarity-biased, perf-biased) + 1 comparator unit `depends_on: both` | `lib/emitters.py::plan_output_to_paired_builders` |

The engine's seam-handler (today: `_maybe_seam`; v0.2.0: generalized) looks up the emitter for the current `(from_phase, to_phase)` tuple in a small dispatch table inside `lib/emitters.py` and calls it. The emitter returns new units; the engine appends them to `units[]` in the SAME atomic write that records the triggering verdict transition (see HLTD-Atomic ordering below). If no emitter exists for a `(from, to)` tuple, the engine raises — explicit beats silent-no-op.

**Test coverage requirement:** each of the four emitters above gets a unit test in `tests/unit/emitters.test.sh` with happy-path + at-least-one-edge-case (e.g. `plan_output_to_paired_builders` when the plan output is empty; `judge_winner_to_work_units` when judge findings don't name a winner).

### Predicate carving (G2 — `met` vs `phase_met`)

Today `exit_predicate_result.met` collapses two semantics: "this phase is done, advance" AND "the run is done, exit." That collapse only holds because v0.1.1 has exactly one work-phase. In A3 with four phases, the collapse fires `met=true` after `work_sketch` returns clean — and the run silent-exits before review/plan/refine ever run.

**Decision: two predicate fields, distinct call sites.**

- `exit_predicate_result.met` retains its meaning: **run-met**. The loop's exit branch reads this. Required: `current_phase == terminal_phase AND <existing scale-aware rules>`. Outside the terminal phase, `met` is always `false`.
- `exit_predicate_result.phase_met` (new field): **current-phase-met**, drives phase advancement. The seam-handler / phase-transition path reads this. Required: the per-phase rules (plan-phase: `gaps_open==0 AND plan_step=="review_plan"`; work-phase variants: scale-aware findings + per-phase `all_units_terminal_in_phase`).
- Both fields recomputed in the same atomic write per I-1. Schema extension is additive — old ledgers read `phase_met` as `null` and the engine treats null as "fall back to met-equals-phase-met behavior" for recipe-blind A1.

**Deliberate-fail test (required, separate from the phase-grammar helper's revert):** Edit `recompute_predicate` to set `met = phase_met` unconditionally (collapse the two semantics). Run the AE5 integration test → asserts the A3 run does NOT silent-exit after sketch. Confirm RED. Restore.

### Atomic snapshot ordering (G3 — emit-then-recompute)

A2's judge unit terminating with `blockers==0 AND majors==0` would trigger `all_units_terminal==true` AND `met==true` under the v0.1.1 rules, EVEN THOUGH the chosen plan's work units haven't emitted yet. Same shape as `feedback_loop_monitor_terminal_state_field`: the terminal-state field is the wrong fingerprint for runs that emit units dynamically.

**Decision: required ordering inside one atomic ledger write.**

1. Apply the triggering transition (e.g., judge unit `verdict-returned`)
2. Run the phase-transition emitter (writes new `pending` units into `units[]`)
3. Recompute `phase_met` AND `met` (now sees the post-emission unit set)
4. Atomic write of the new ledger

If steps 2 and 3 swap, A2 silent-exits before work units appear. If step 2 isn't in the same atomic snapshot, a crash between transition and emission leaves the ledger in an exited-looking-but-incomplete state — un-resumable cleanly.

**New ledger I-1 corollary** (documented in U11's contract update): "Phase-transition emission writes new pending units in the same atomic snapshot as the verdict that triggers it; predicate recomputation runs on the post-emission unit set." This is parallel to today's atomic predicate freshness invariant.

**Deliberate-fail test:** Edit the seam-handler to recompute predicate BEFORE running the emitter on an A2 ledger fixture. Assert the predicate fires `met=true` after judge (incorrect). Confirm RED of the AE4 integration test. Restore.

### Scalar mirror under N parallel plan-loops (G4 — pick serialized advance, defer adapter re-lock)

The original plan posited "engine writes scalar `plan_step` mirror of currently-advancing plan-loop unit" for adapter backward compatibility under N>1 plan units. This is incoherent under truly concurrent advance — with 3 plan units potentially at 3 different `plan_step` values, the scalar holds ONE value, and the adapter reads inconsistent state.

**Decision: plan-loop units in N>1 mode execute SERIALIZED — one `adapter.next_plan_step` call per tick, round-robin across ready plan units.** The orchestrator marks a single plan unit as "actively advancing" each tick; the scalar `plan_step` mirrors that unit's value during its tick. Other plan units are dispatched-but-idle (their `state` is `dispatched`, but they're not mid-adapter-call).

This preserves AE4's spirit ("3 units in `dispatched` immediately after first tick") with the precise interpretation: dispatched from the orchestrator's perspective; the adapter sees one at a time.

**Deferred to v0.3.0 (not v0.2.0):** re-locking the adapter contract to add a `unit_id` or `unit_view` parameter to `next_plan_step` so parallel adapter calls become possible. V1's serialized advance is correct and simple; concurrent advance is an optimization with a contract cost.

**KTD-4 (below) restates this as a Key Technical Decision** for surface visibility.

### Start phase propagation

Two notes for the helper:

- `init_ledger` sets `loop_phase=recipe["phase_order"][0]`. For A1/A2/A4 this is `"plan"` (unchanged from v0.1.x). For A3 this is `"work_sketch"` — the engine MUST NOT assume "plan is first."

## System-Wide Impact

- **Interaction graph:** Adding recipes touches the engine's loop construction (`init_ledger`), tick loop (`_tick_body`, the former `_maybe_seam` now `_advance_phase_if_met`, `advance_plan_loop`), predicate evaluation (`recompute_predicate` — now writes two fields per G2), status reporting (`auto-status`), and resume (`auto-resume`). All except `auto-resume` read the new ledger fields directly. `auto-resume` re-reads the ledger fresh on suspend recovery, so it picks up recipe state automatically — no change needed.

- **Error propagation:** Recipe validation errors at engine load (`load_and_validate`) propagate to `lib/auto.py::run` which exits 1 with a clear message. Recipe validation errors at skill-write propagate to the skill which surfaces them to the user and refuses to write. The contract is `validate` (hard errors, used by both engine and skill) + `validate_and_lint` (additional editorial checks, skill only) per KTD-2. Same hard-error shape across both callers; lint warnings are skill-only and never reject a contract-valid recipe.

- **State lifecycle risks:** A v0.1.x ledger MUST be readable as a v0.2.0 ledger (additive-extension invariant). The R13 regression test enforces this. A run started under v0.2.0 with a workspace recipe survives the recipe file being deleted mid-run — `auto-resume` reads the ledger, not the recipe file, so topology info carries forward via the ledger's `recipe`/`phase_order`/`terminal_phase` fields (KTD-12). New atomic-ordering invariant (G3): phase-transition emission writes new pending units in the same atomic snapshot as the verdict that triggers it; a crash between transition and emission is impossible by construction.

- **API surface parity:** `--recipe` is a new CLI flag. The `--adapter` flag is orthogonal and unchanged. The picker is a new UX surface on `commands/auto.md`. The authoring skill is a new skill. No existing command, flag, or behavior is removed or changed.

- **Integration coverage:** AE tests distributed across owning units (AE1+AE2 → U8, AE3 → U9, AE4 → U5b, AE5 → U5). Each AE has a deliberate-fail check verified via the Edit tool. The four built-in recipes are the validator's conformance corpus. The R13 regression test (A1 ≡ v0.1.1) is the recipe-pipeline-wired-correctly smoke test. U10 is the verification gate that audits all deliberate-fail checks before merge.

- **Unchanged invariants:**
  - `dispatch_batch`, `ready_units`, `_is_ready`, `_dependency_satisfied`, `unit_is_terminal`, `record_verdict`, `transition`, `_atomic_write`, `_with_locked_ledger`, `advance_work_loop`, `_ready_fix_unit`, `_ready_reenqueue_unit`, `detect_and_halt_stalled` — all topology-agnostic, all unchanged.
  - The adapter contract (`docs/contracts/adapter-contract.md`) signature set — unchanged in V1 (KTD-4: plan-loop advance is serialized so the adapter sees one unit at a time; concurrent advance + adapter-contract re-lock deferred to v0.3.0).
  - The smoke-test `$ARGUMENTS`-uniqueness invariant on `commands/*.md` — unchanged. Picker logic adds prose only.
  - The state grammar (`pending → dispatched → ... → terminal-*`) — unchanged.
  - The I-1 atomic predicate freshness invariant — preserved AND extended (G3 corollary: emit-before-recompute inside the same atomic write).

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| **Plan documents R11/R12 behavior the code doesn't fully wire** (dominant build-bug class from v0.1.1) | Centralized `lib/phase-grammar.py` helper as single decision point; AST lint test enforces no literal `loop_phase` reads outside the helper (KTD-3 — mechanical, not convention); three deliberate-fail tests in U5 each break a different drift mode. |
| **G1 — Dynamic unit emission silently broken** (A2/A3/A4 ship as validating JSON but units never appear at runtime) | Explicit named-function registry in `lib/emitters.py` (U5b, KTD-5); each emitter has unit tests + at least one edge case; AE4 integration test asserts the dynamic-emission path produces work units; atomic-ordering deliberate-fail confirms emit-then-recompute is correct. |
| **G2 — Predicate semantic conflation** (A3 silent-exits after sketch because `met` is overloaded for both phase-advance and run-exit) | Two distinct fields in `exit_predicate_result`: `met` (run-exit, terminal-phase-only) and `phase_met` (advance, all phases). Deliberate-fail test reverts the carving (`met = phase_met` unconditionally) and asserts AE5 goes RED. |
| **G3 — Predicate fires before dynamic units emit** (A2 judge terminal → ledger has zero work units → `all_units_terminal==true` → silent exit) | New ledger I-1 corollary: phase-transition emission writes new pending units in the same atomic snapshot as the triggering verdict. Order inside the write: (1) apply transition, (2) run emitter, (3) recompute predicate, (4) atomic write. Deliberate-fail test reverses 2 and 3, asserts AE4 RED. |
| **G4 — Scalar mirror under concurrent N>1 plan advance is structurally broken** | Resolved by KTD-4: plan-loop units in N>1 mode execute SERIALIZED (round-robin one-at-a-time per tick). Adapter contract unchanged in V1. Concurrent advance + contract re-lock deferred to v0.3.0. |
| **AskUserQuestion preview field doesn't render multi-line ASCII** | V1 fallback is short labels + descriptions; multi-line ASCII is an upgrade attempted in implementation; if it doesn't render, ship the fallback. Both paths tested in U8. |
| **Three independent ASCII renderers drift** (picker, skill, status) | Single renderer at `lib/topology-render.py` (KTD-10, U7); called from all three surfaces. Visual-vocabulary doc is the renderer's contract, not a separate spec. |
| **Authoring skill and engine loader validate differently** | `validate` is the shared contract (called by both); `validate_and_lint` is the skill-side editorial layer that adds non-blocking warnings (KTD-2). Test asserts skill output that passes `validate` is engine-acceptable. |
| **A1 recipe diverges from v0.1.1 behavior** | R13 regression test ("first-tick ledger from `--recipe a1` ≡ first-tick ledger from v0.1.1 invocation, modulo additive fields") is the wiring-correctness smoke test. Run on every PR. Note: removed the legacy hardcoded path per KTD-1 — no organic diff-test safety net during transition. |
| **Round-1 fixes only the cited site (narrowing class)** | AST lint test (KTD-3) enforces structural single-source-of-truth mechanically; review-loop budget allocated for ≥3 rounds per institutional learning. |
| **Backward-compat breaks on a v0.1.x ledger** | U1 test loads a pinned v0.1.x fixture and asserts predicate evaluation matches v0.1.1's computation. Fail-stop if it diverges. |

## Phased Delivery

Two phases — engine + format + emitters first, picker UX + skill second. Each phase is an internally-coherent landing point that doesn't ship a half-feature. Per KTD-1, Phase 1 removes v0.1.x's legacy hardcoded `init_ledger` path; bare `/auto <plan>` defaults to A1 via the recipe pipeline from Phase 1 onward, before the picker ships in Phase 2.

### Phase 1 — Engine, format, emitters (lands first)
- U1 (ledger schema extension — additive)
- U2 (recipe format + JSON Schema validator + recipe-format.md contract)
- U3 (loader + 3-tier registry + `unit_for` + `validate_and_lint`)
- U7 (four built-in recipes + `lib/topology-render.py`)
- U5 (phase-grammar helper + predicate carving + AST lint)
- U5b (phase-transition emitter registry + atomic-ordering wire-up)
- U6 (serialized N-parallel plan-loop advance)
- U4 (`--recipe` flag + replace v0.1.x hardcoded init_ledger path with recipe pipeline; A1 is the new default)
- AE4 (in U5b), AE5 (in U5), R13 regression (in U4) integration tests

**Landing point:** `--recipe <name>` works end-to-end. Bare `/auto <plan>` runs A1 by default (via the recipe pipeline, NOT a legacy path — KTD-1). The four built-ins run via the explicit flag. Picker hasn't shipped yet; users must type `--recipe a2` etc. to pick non-default. AE4 and AE5 pass; R13 regression passes; all v0.1.x tests pass unchanged.

### Phase 2 — Picker UX + authoring skill + docs
- U8 (picker in commands/auto.md + recipes-list.sh + recipe-picker.test.sh)
- U9 (authoring skill + recipe-authoring.test.sh)
- U10 (verification gate: deliberate-fail audit across all of Phase 1 + Phase 2)
- U11 (docs + version bump + new emitter-registry.md contract)

**Landing point:** Bare `/auto <plan>` opens the picker. `/auto-author-recipe` is available. AE1/AE2/AE3 pass. v0.2.0 is feature-complete.

If pressed for time, Phase 1 alone is a meaningful v0.2.0-alpha — the recipe format + four built-ins + explicit `--recipe` invocation lands the architectural unlock. Phase 2's picker and skill are the polish, but the picker is what makes recipes discoverable to non-power-users.
