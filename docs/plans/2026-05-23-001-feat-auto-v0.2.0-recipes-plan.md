# feat: auto v0.2.0 — Recipes (packaged workflows for the loop engine)

## Overview

`auto` v0.2.0 introduces **recipes** — named, file-backed JSON declarations of workflow topology. A recipe describes the initial ledger the engine builds (which units exist, their `depends_on` graph, which phase each runs in, and what each invokes). The engine reads the recipe and builds the ledger; everything downstream of that (tick loop, dispatch, predicate evaluation, resume) runs unchanged for A1 and extended for non-default topologies.

v0.2.0 ships:

1. The **JSON recipe format** + a single validator used by both the engine loader and the authoring skill.
2. **Three built-in recipes** (A1/A2/A4) encoding workflows the user already runs manually. (A3, Build-First Feedback, is deferred to v0.2.1 — it alone drives ~40% of the engine surface; see KTD-14.)
3. A **three-tier registry** (workspace → global → built-in) with first-wins resolution.
4. A **picker** at `/auto` start via `AskUserQuestion`.
5. An **authoring skill** (`auto-author-recipe`) that compiles recipes from prose.
6. **Engine extensions**: R11 (multiple parallel plan-loops, serialized advance) ships in v0.2.0; R12 (non-default phase ordering) ships its field-reservation only — the runnable form is deferred to v0.2.1 with A3. The adapter contract re-locks once, adding `enumerate_plan_units`.

Behavioral guarantee: v0.1.x users see no behavioral change. A1 is the canonical encoding of v0.1.1's behavior — picking A1 in the picker (or accepting the default) produces a first-tick ledger byte-identical to v0.1.1's.

---

## Problem Frame

v0.1.1 ships exactly one workflow shape — plan → seam → work → exit-on-P3 — and the user already runs at least four distinct workflows manually using Claude primitives + popular plugins. v0.2.0 packages those workflows as recipes so `auto`'s durable ledger, fan-out, and resume can serve all four. The frame, from the brainstorm: ***`auto` is just packaging Claude primitives and popular workflow plugins.*** (see origin: `docs/brainstorms/2026-05-23-auto-v0.2.0-recipes-requirements.md`)

## Context & Research

### Relevant Code and Patterns

- **The single change point for recipe→ledger:** `lib/auto.py:194` — `run()` calls `ledger.init_ledger(repo_root, run_id, adapter=adapter, units=[], loop_phase="plan")`. Today `units=[]` and `loop_phase="plan"` are hardcoded. Recipes replace those two argument values. `init_ledger` (`lib/ledger.py:489`) already accepts arbitrary `units` and `loop_phase`; the caller just never supplies them.
- **Phase grammar enforcement (the R12 surface):** Three load-bearing sites all key on the literal `"plan"` vs `"work"` distinction:
  - `lib/tick.py:533-562` — `_tick_body` 3-way phase switch
  - `lib/tick.py:634-671` — `_maybe_seam` hardcodes plan→seam(→work) transition
  - `lib/ledger.py:315-352` — `recompute_predicate` binary plan-vs-everything-else branch
  - Plus secondary touchpoints at `lib/tick.py:508` (predicate-met preempt) and `lib/tick.py:610` (post-advance recheck).
- **Plan-loop singularity (the R11 surface):** Enforced by three top-level scalars + one adapter call: `ledger.plan_step` and `ledger.exit_predicate_result.gaps_open` (scalars, one per run); `lib/tick.py:376 advance_plan_loop` calls `adapter.next_plan_step(ledger_dict)` which returns ONE step; today `units=[]` in plan phase so dependency machinery is dormant.
- **Engine seam (unchanged in v0.2.0):** `lib/orchestrator.py::dispatch_batch`, `ready_units`, `_is_ready`, `_dependency_satisfied`, `_ancestor_ids`, `converge`; `lib/ledger.py::unit_is_terminal`, `record_verdict`, `transition`, `_atomic_write`, `_with_locked_ledger`; `lib/tick.py::advance_work_loop`, `_ready_fix_unit`, `_ready_reenqueue_unit`, `detect_and_halt_stalled`. These read only `units[].state`/`depends_on`/`findings[]`/`adapter_scale` and stay topology-agnostic.
- **Commands→lib dispatch pattern:** Three commands today (`auto.md/auto-status.md/auto-resume.md`) each with NL routing prose in the .md and one canonical `bash "${CLAUDE_PLUGIN_ROOT}/lib/X.sh" "$ARGUMENTS"` dispatch line. The smoke test (`tests/smoke/scaffold.test.sh:87-109`) enforces this exactly-once invariant. The picker logic for R7 lives entirely in `commands/auto.md`'s NL routing prose — no new dispatch lines, no smoke-test changes.
- **Test harness:** `tests/run.sh` collects `tests/{smoke,unit,integration}/*.test.sh`. Drop a new `tests/unit/recipes.test.sh` and it auto-runs. Helpers at `tests/helpers/test-helpers.sh`. Pattern examples: `tests/unit/ledger.test.sh` (875 lines), `tests/unit/orchestrator.test.sh` (640 lines).
- **Plugin manifest auto-globs:** `.claude-plugin/plugin.json` declares `skills: "./skills"` and `commands: "./commands"`. Adding `skills/auto-author-recipe/SKILL.md` requires no manifest edit.
- **Contracts (locked, additive-only extension convention):** `docs/contracts/ledger-schema.md` already exercises the additive-extension pattern (the `attempt` field from Bug #6 — schema §2.3 documents `attempt: 0` default and notes "additive / backward-compatible — an old ledger with no `attempt` field reads as `0`"). v0.2.0 uses the same shape for all new fields.

### Institutional Learnings

- **`feedback_plan_documents_transition_code_doesnt_wire_it`** is the anchor learning for this plan. R11 and R12 are exactly the "prose-says-X, engine-must-now-honor-X" shape that bit v0.1.1 six different ways. Every recipe topology rule must be wired through a single centralized helper, not pattern-matched at each consumer. Allocate review-loop budget to post-build code review, not pre-build doc deepening.
- **`feedback_fix_the_class_not_the_cited_instance`**: grep the v0.1.1 engine for every site that branches on `loop_phase` (predicate, `_tick_body`, `_maybe_seam`, status reporting, resume). All sites route through the new phase-grammar helper. Round-1 fixing only cited sites is a known failure mode.
- **`reference_claude_loop_task_daemon_primitives`**: each background agent self-writes its verdict to the ledger atomically; A2's judge unit reads verdicts from disk, not from in-memory `backgroundTasks`. The same pattern applies to A4's comparator.
- **`feedback_slash_command_nl_routing_pattern`** + **`feedback_slash_command_arg_substitution`**: picker logic and `--recipe` flag parsing live in `commands/auto.md`'s NL routing prose; `lib/auto.sh` stays headless. The .md body keeps the exactly-one-`$ARGUMENTS`-line invariant.
- **`feedback_loop_monitor_terminal_state_field`**: A3 has two work-phases. Without a `terminal_phase` field, the predicate fires after the sketch work-phase and the run silently exits. Required schema field for any non-default `phase_order`.
- **`feedback_predicted_bugs_need_tests_not_conventions`** + **`feedback_deterministic_over_probabilistic_v1`**: validation is mechanical, not conventional. JSON Schema validator gates the recipe at load AND at skill-write time. **Accept the reserved `python_hook` field but ignore it; reject ALL OTHER unknown top-level fields.** (Per R3, `python_hook` is a future-compat reservation, not an error.) The "reject unknown fields" principle holds for everything outside the explicit reserved list.
- **`feedback_new_tests_need_deliberate_fail_smoke_check`**: every new test (AE1–AE4 + the A4 runtime test + schema validator tests + the V1-rejection test for non-default phase_order) gets a deliberate-fail check via the **Edit tool**, not a sed/regex revert. Particularly load-bearing for the A4 runtime test (the `(plan,work)` emitter-disambiguation could silently fire A1's emitter instead — adversarial F2) and the atomic-ordering test (A2 could silent-exit before work units emit). (AE5's sketch-fires-first deliberate-fail moves to v0.2.1 with A3.)
- **`feedback_subagent_write_verification`**: the authoring skill, after writing the recipe file, reads it back and runs the validator before reporting "saved." Same for any subagent-dispatched implementation units in this plan.
- **`feedback_review_loop_catches_narrowing`**: budget ≥3 code-review rounds. Claude-modes V2 took 9; claude-dispatch v0.1.1 took 3+.

### External References

None. Pure Python 3 stdlib + bash + on-disk JSON; no framework dependencies. The locked contracts in `docs/contracts/` are the canonical reference.

## Key Technical Decisions

- **KTD-1. A1 is a Python constant; `recipes/a1.json` is the user-facing override target.** *(Round-2 revision per adversarial F4 — the original "remove legacy path, A1 always from JSON" choice introduced a new runtime file dependency on every bare `/auto` invocation.)* The canonical A1 topology lives in `lib/recipes.py` as `A1_BUILTIN = {...}`. The picker and registry resolution still surface `recipes/a1.json` if present in workspace/global tiers (override path), but if absent at the built-in tier, the engine falls back to the constant. **R13 regression test** ("first-tick ledger from A1 recipe ≡ v0.1.1's first-tick ledger, modulo additive fields") still holds — it tests against whichever source resolves. **Trade:** one canonical source eliminates drift; built-in install corruption can't break bare `/auto`; user-facing override still works.
- **KTD-2. Base validator + optional lint pass.** The JSON Schema validator lives in `lib/recipes.py` as `validate(recipe_dict)`. The authoring skill calls a wrapping `validate_and_lint(recipe_dict)` that adds editorial checks the engine ignores: "phase declared in `phase_order` but no unit assigned to it," "depends_on creates an unreachable subgraph," "terminal_phase has no units assigned," **"workspace/global recipe description text matches a built-in's verbatim"** (added per security observation 1 — defense against description-spoofing). Both layers live in `lib/recipes.py`; the contract is `validate`, the editorial pass is `lint`. Skill output that passes `validate` is engine-acceptable; skill UX gets quality-error catching at write time.
- **KTD-3. Phase-grammar helper enforcement: AST lint catching the STRING CLASS, not just constant-subscript shape.** *(Strengthened per adversarial F5 — the round-1 design was regex-strength dressed as mechanical.)* Every site that today branches on `loop_phase` reads from `lib/phase-grammar.py::current_phase(ledger)`. The lint forbids the **string literal `"loop_phase"`** anywhere outside `lib/phase-grammar.py` (not just `ledger["loop_phase"]` subscripts). All identified bypasses (`_key = "loop_phase"; ledger[_key]`, dict iteration, `getattr`, `__getitem__`) require the string to exist somewhere — banning the string literal catches the class. Mechanical, not convention.
- **KTD-4. Plan-loop units in N>1 mode execute SERIALIZED. Adapter contract re-locks with new op `enumerate_plan_units`.** *(Round-2 revision per feasibility F4 — resolves both G4 AND the producer gap.)* The orchestrator advances exactly one plan-loop unit per tick (serialized round-robin) — so the existing `next_plan_step(ledger)` signature is preserved unchanged for that op. **A new adapter op `enumerate_plan_units(ledger) → list[unit_dict]` is added to the locked contract.** Both `ce` and `native` adapters implement it. The op runs at `plan-done` and returns the list of work units the plan produced. The emitter `plan_output_to_work_units` reads this list via the adapter call, not from a ledger field. **This is the v0.2.0 adapter contract re-lock** — required because the previous "scalar mirror handles compat" framing was incoherent AND the emitter abstraction had no source data. Round-robin tie-breaker: `units[]` declaration order, lowest-index ready plan unit with the smallest per-unit `plan_step` value wins next tick.
- **KTD-5. Phase-transition emitters are an explicit named-function registry in `lib/emitters.py`; recipes don't declare emitter names in V1.** (Resolves G1.) The engine maps `(from_phase, to_phase)` tuples to emitter function names. V1 ships **exactly 3 emitters** (A3 deferred per KTD-14 below, so the `review_findings_to_plan_input` emitter is also deferred): `plan_output_to_work_units` (A1 + A2's chosen-plan downstream), `judge_winner_to_work_units` (A2), `plan_output_to_paired_builders` (A4). The implicit-vs-explicit fork: rejected "recipes declare emitter names" — engine ships the small set; v0.3.0 can add a recipe-declared field if the implicit set becomes too narrow.
- **KTD-6. Atomic ordering via new `transition_and_emit` ledger primitive.** *(Round-2 revision per feasibility F2 + adversarial F1 — round-1's "same `_with_locked_ledger` write" framing was implementable only with a NEW primitive that the plan never specified.)* The seam-handler does NOT call `record_verdict` then a second write for emission. Instead, the orchestrator detects (on the next tick) that a triggering unit transitioned (e.g., judge verdict-returned in a prior tick's `record_verdict`) and calls `ledger.transition_and_emit(emitter_fn, ...)` — a single new `_with_locked_ledger` body that reads ledger, calls emitter, appends pending units, recomputes predicate, atomic-writes. **Emitters MUST be pure functions** (signature `emit(ledger, triggering_unit) → list[new_unit_dict]`); they MUST NOT call `record_verdict`/`transition`/`set_loop` (re-entrancy: `_flock_run` opens a new fd each call → deadlock). The seam-handler owns the lock; the emitter only computes.
- **KTD-7. Per-unit `plan_step` and `gaps_open` for plan-phase units (R11).** A1's single plan-loop continues to use the top-level scalars for byte-identical compatibility. A recipe with N plan-loops uses per-unit fields. The `recompute_predicate` plan branch reads per-unit when present, scalar otherwise. **Combined with KTD-4's serialized advance, the per-unit fields are written one at a time (the actively-advancing unit's plan_step gets updated each tick).**
- **KTD-8. Single predicate field `met`, phase-aware.** *(Round-2 revision — KTD-6 predicate carving from round 1 was driven solely by A3's two work-phases. With A3 deferred per KTD-14, the carving is unnecessary.)* `met` retains today's meaning, extended to be aware of `phase_order` and `terminal_phase` (when present): in `loop_phase == "plan"`, `met` requires existing plan-rules; in `loop_phase == terminal_phase`, `met` requires existing work-rules. With A1/A2/A4 all using default `phase_order = ["plan", "seam", "work"]` and `terminal_phase = "work"`, the predicate's behavior is byte-identical to v0.1.1's for A1; A2's parallel plan-loops use the per-unit aggregation path from KTD-7. The two-field carving (`phase_met`) is **deferred to v0.2.1 with A3**.
- **KTD-9. Picker default: short-label options with one-line descriptions.** Multi-line ASCII previews via `AskUserQuestion`'s preview field land as the picker's full form **if** verified to render. Otherwise, the picker uses short labels and includes a hint "type `/auto --recipe a1` to skip the picker." A `/auto --show-topology a1` command (deferred to implementation) is the fallback surface for visual reference.
- **KTD-10. Single ASCII topology renderer at `lib/topology-render.py`, called by picker + authoring skill + auto-status.** The renderer takes `(recipe_dict, width_hint) → str`. `skills/auto-author-recipe/references/visual-vocabulary.md` is the renderer's contract doc, not a separate spec.
- **KTD-11. `recipe` and `phase_order` are top-level ledger fields.** Schema extension is additive: missing `recipe` → recipe-blind v0.1.x ledger (treated as A1); missing `phase_order` → default `["plan", "seam", "work"]`.
- **KTD-12. Resume re-reads the ledger, not the recipe file.** The ledger's `recipe` + `phase_order` fields carry forward all topology info; the recipe file is only read at `init_ledger` time. Resume across recipe file deletion is safe.
- **KTD-13. `--recipe` flag writes resolved tier to stderr when non-built-in.** *(Added per security finding 2 — supply-chain hijack via workspace recipe shadowing.)* When `resolve(name)` returns a non-built-in tier, the resolver emits `[auto] resolving 'a1' from workspace (.claude/auto/recipes/a1.json)` to stderr. CI logs show the override; interactive users in the picker already see the tier badge. Low cost, closes the silent-shadow attack surface.
- **KTD-14. A3 (Build-First Feedback) deferred to v0.2.1.** *(Round-2 revision per product-lens Finding 1 — A3 alone drives ~40% of v0.2.0's engine surface: predicate carving, `terminal_phase` requirement, `review_findings_to_plan_input` emitter, non-default `phase_order` support, AE5. With A3 deferred, v0.2.0 ships A1+A2+A4 as the recipe-format proof points and the picker has 3 distinct cards instead of 4. The brainstorm's original "all four together" decision was made WITHOUT visibility into how concentrated A3's engine cost would be — that surfaced only at round-1 deepening.)* v0.2.1 gets its own brainstorm/plan/review cycle for A3 once R11's parallel-plan mechanics have proven out in the field. The recipe format reserves `phase_order` and `terminal_phase` fields as future-compat — they parse but the V1 validator REJECTS values other than the default for non-A3 recipes (no untested code paths ship). Tests against A3's specific shape (AE5, the two-field predicate, the `review_findings_to_plan_input` emitter) are removed from v0.2.0.

## Open Questions

### Resolved During Planning

- **Q1 — Ledger schema extension shape**: Additive top-level fields (`recipe`, `phase_order`, `terminal_phase`); additive per-unit fields (`phase`, `plan_step`, `gaps_open`, `dispatch_context`, `last_advanced_at`). All default such that v0.1.x ledgers are A1-equivalent. (`last_advanced_at` added per feasibility F5 — the round-robin tiebreaker discriminator; `dispatch_context` per feasibility F-B — the recipe-side `prompt_template` merge target.)
- **Q2 — Topology declarations for A1/A2/A4**: See Implementation Unit U7 for concrete JSON. (A3 deferred per KTD-14.)
- **Q3 — Adapter contract change for R11 (resolves the producer gap)**: **V1 re-locks the adapter contract** with one new op: `enumerate_plan_units(ledger) → list[unit_dict]` (per KTD-4 / feasibility F4). Plan-loop units in N>1 mode execute SERIALIZED — `next_plan_step(ledger)` signature is preserved; `enumerate_plan_units` is the producer the emitters read. Both `ce` and `native` adapters implement the new op. Concurrent plan advance (a `unit_id` parameter on `next_plan_step`) stays deferred to v0.3.0.
- **Q4 — Phase-aware exit predicate**: Single `met` field, phase-aware (per KTD-8). A1/A2/A4 all use default `phase_order`/`terminal_phase`, so the predicate is byte-identical to v0.1.1 for A1 and uses per-unit aggregation for A2. The two-field carving (`phase_met`) is **deferred to v0.2.1 with A3** (the only recipe that needed it).
- **Q5 — Picker on no-resolvable recipes**: `--recipe nonexistent` → exit 1 with resolved set printed. Built-in A1 absent → falls back to the `A1_BUILTIN` Python constant (per KTD-1); other built-ins absent → exit 1 with reinstall hint.
- **Q6 — Dynamic unit emission (G1)**: Explicit named-function registry in `lib/emitters.py`. V1 ships **3 emitters** (A3's `review_findings_to_plan_input` deferred per KTD-14): `plan_output_to_work_units`, `judge_winner_to_work_units`, `plan_output_to_paired_builders`. The engine maps `(from_phase, to_phase)` tuples to emitter names. See HLTD "Dynamic unit emission" section for the full table. The emitters read plan output via the new `enumerate_plan_units` adapter op (Q3).
- **Q7 — Atomic ordering via new ledger primitive (resolves G2/G3 implementability)**: A new `ledger.transition_and_emit(emitter_fn, ...)` primitive does the transition + emit + recompute + atomic-write in ONE `_with_locked_ledger` body (per KTD-6 / feasibility F2 / adversarial F1). Emitters are pure functions that MUST NOT call ledger mutators (re-entrancy / deadlock — `_flock_run` opens a new fd each call). The seam-handler owns the lock.
- **Q8 — `_bootstrap.py` shim does not exist; U5 must create it**: per feasibility F1 / repo-research F-A / adversarial F6, `lib/_bootstrap.py` today only has `load_ledger()`. Hyphenated modules (`adapter-ce.py`) are loaded inline in `lib/tick.py::resolve_adapter`. U5 adds a generic `load_lib_module(name)` to `_bootstrap.py` for `phase-grammar.py` (and reusable for `topology-render.py`). This is a real new shim, not a phantom one.

### Deferred to Implementation

- **The exact AskUserQuestion preview surface for ASCII multi-line content.** First implementation pass tries the preview field with multi-line ASCII; if rendering breaks, falls back to short labels per KTD-9. The fallback is the default behavior; the multi-line preview is the upgrade. Test both; ship whichever renders.
- **Exact prose-elicitation flow for the authoring skill (R9).** Sketched in U9; refined during implementation by running the skill against the three V1 built-in topologies (A1/A2/A4) as test cases (AE3 is the conformance test).
- **Whether the judge's verdict-write convention in A2 needs a new adapter op or can reuse `record_verdict`.** Initial pass: judge unit uses `record_verdict` like any other unit; its findings array carries the chosen winner (which the `judge_winner_to_work_units` emitter reads to know which plan's output to emit via `enumerate_plan_units`). If that's too restrictive in practice, surface as v0.3.0 work.
- **The exact `emits`/`emitter` field shape on recipe-declared units, when v0.3.0 lifts the implicit-emitter restriction.** V1 ships 3 hardcoded emitters with a per-phase-transition `emitter` hint disambiguating A1 vs A4; v0.3.0 may let recipes declare arbitrary custom emitters. The full contract for arbitrary emitters is deferred until a user-authored recipe needs one.

### Recipe JSON shape (canonical)

A1 (Classic CE Stack) as the simplest concrete instance:

```jsonc
{
  "name": "a1",
  "version": "1",
  "description": "Classic CE Stack — plan → build → review → fix to P3-only exit",
  "phase_order": ["plan", "seam", "work"],
  "terminal_phase": "work",
  "default_adapter": "ce",
  "units": [
    {
      "id": "plan",
      "phase": "plan",
      "depends_on": [],
      "invokes": {"adapter_op": "next_plan_step"}
    }
    // work units appear dynamically — they're emitted by the plan-loop into units[]
    // at runtime; the recipe doesn't pre-declare them because plan output produces them
  ]
}
```

A2 (Parallel Theories + Judge):

```jsonc
{
  "name": "a2",
  "phase_order": ["plan", "seam", "work"],
  "terminal_phase": "work",
  "units": [
    {"id": "plan-1", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}},
    {"id": "plan-2", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}},
    {"id": "plan-3", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}},
    {"id": "judge",  "phase": "work", "depends_on": ["plan-1", "plan-2", "plan-3"],
     "invokes": {"adapter_op": "review", "prompt_template": "judge.md"}}
    // work units from chosen plan emitted dynamically after judge resolves
  ]

```

A3 (Build-First) is **deferred to v0.2.1** per KTD-14 — it's the only recipe needing non-default `phase_order`, and it drove ~40% of v0.2.0's engine surface. Its sketch is preserved in the v0.2.1 brainstorm. The recipe format reserves `phase_order` and `terminal_phase` as future-compat fields; the V1 validator rejects non-default values so no untested phase-ordering code path ships.

A4 (Adversarial Pair + Comparator):

```jsonc
{
  "name": "a4",
  "phase_order": ["plan", "seam", "work"],
  "terminal_phase": "work",
  "units": [
    {"id": "plan", "phase": "plan", "depends_on": [], "invokes": {"adapter_op": "next_plan_step"}}
    // Two builders + comparator emitted dynamically once plan resolves, OR pre-declared:
    // {"id": "build-clarity", "phase": "work", "depends_on": [], "invokes": {..., "bias": "clarity"}},
    // {"id": "build-perf",    "phase": "work", "depends_on": [], "invokes": {..., "bias": "perf"}},
    // {"id": "compare", "phase": "work", "depends_on": ["build-clarity", "build-perf"],
    //  "invokes": {"adapter_op": "review", "prompt_template": "compare.md"}}
  ]
```

**Note on `invokes` field shape.** Each unit's `invokes` dict carries `adapter_op` (one of the locked ops in `docs/contracts/adapter-contract.md` §2: `plan`, `deepen`, `review_plan`, `next_plan_step`, `do_unit`, `review`, plus the v0.2.0 re-lock addition `enumerate_plan_units`) plus optional recipe-side metadata like `prompt_template`. The recipe-side metadata IS represented on the ledger via the additive `units[].dispatch_context` field (per U1) — the recipe loader (`lib/recipes.py::unit_for`) merges `prompt_template` etc. into `dispatch_context` BEFORE dispatch, after path-bounding validation. The adapter reads `dispatch_context` via its existing `unit` parameter.

### Dynamic unit emission (G1 — phase-transition emitters)

The `// emitted dynamically` comments are the single most important wiring point in v0.2.0 — if hand-waved, A2/A4 ship as JSON that validates but silent-fails at runtime. This subsection is the explicit specification.

**The producer (resolves feasibility F4 — the gap that almost shipped):** v0.1.1 does NOT produce work units in code — `next_plan_step` returns one step token and the seam pauses for manual work-unit creation. So the emitters have no source data unless the adapter provides it. **v0.2.0 re-locks the adapter contract with `enumerate_plan_units(ledger) → list[unit_dict]`** (KTD-4). The plan-loop, on reaching `plan-done`, calls this op; the emitter reads its return value. Both `ce` and `native` adapters implement it.

**Decision: emitters are an explicit registry, recipe-blind for V1.** A new module `lib/emitters.py` owns a registry of named emitter functions. The engine maps `(from_phase, to_phase)` tuples to emitter names (the "trigger" column documents which verdict transition fires that emitter — it's not part of the dispatch-table key). V1 ships exactly the **3** emitters the V1 built-ins need (A3's `review_findings_to_plan_input` deferred with A3 per KTD-14).

Phase-transition emitter table for the V1 built-ins (A1, A2, A4):

| Recipe | Phase boundary | Trigger | Emitter | New unit shape | Owning function |
|---|---|---|---|---|---|
| A1 | `plan` → `seam` → `work` | plan-loop-met | `plan_output_to_work_units` | `enumerate_plan_units` output → 1 unit per item | `lib/emitters.py::plan_output_to_work_units` |
| A2 | judge unit `terminal-no-blockers` → `work` (chosen) | judge verdict-returned | `judge_winner_to_work_units` | judge findings carry the winning plan-loop unit id; emit the winner's `enumerate_plan_units` output as work units | `lib/emitters.py::judge_winner_to_work_units` |
| A4 | `plan` met → `work` | plan-loop-met | `plan_output_to_paired_builders` | `enumerate_plan_units` output → 2 work units (clarity-biased, perf-biased via `dispatch_context.bias`) + 1 comparator unit `depends_on: both` | `lib/emitters.py::plan_output_to_paired_builders` |

**Disambiguation (resolves adversarial F2):** A1 and A4 share the `(plan, work)` tuple. The dispatch table is keyed by `(from_phase, to_phase, recipe_name)` for these collisions, OR the recipe carries an explicit `emitter` hint per phase-boundary. **Decision: recipe carries an `emitter` field per phase-transition** (a narrow exception to "recipes don't declare emitter names" — needed because A1 and A4 are structurally distinct at the same phase boundary). This is the minimum disambiguation; it's not the full "recipes declare arbitrary emitters" v0.3.0 feature.

The seam-handler calls `ledger.transition_and_emit(emitter_fn, ...)` (the new primitive from KTD-6) which looks up the emitter, calls it, appends units, recomputes predicate, atomic-writes — all in one locked body. If no emitter exists for a boundary, the engine raises.

**Test coverage requirement:** each of the 3 emitters gets a unit test in `tests/unit/emitters.test.sh` (happy-path + edge case). **PLUS** each recipe gets an end-to-end integration test asserting the RIGHT emitter fires for its boundary (resolves adversarial F2 — A4 must have its own runtime test, not just A1's emitter reused): AE4 covers A2, a new A4 integration test covers `plan_output_to_paired_builders`.

### Atomic snapshot ordering (G3 — emit-then-recompute)

A2's judge unit terminating with `blockers==0 AND majors==0` would trigger `all_units_terminal==true` AND `met==true` under the v0.1.1 rules, EVEN THOUGH the chosen plan's work units haven't emitted yet. Same shape as `feedback_loop_monitor_terminal_state_field`: the terminal-state field is the wrong fingerprint for runs that emit units dynamically.

**Decision: required ordering inside one atomic ledger write.**

1. Apply the triggering transition (e.g., judge unit `verdict-returned`)
2. Run the phase-transition emitter (writes new `pending` units into `units[]`)
3. Recompute `met` (now sees the post-emission unit set)
4. Atomic write of the new ledger

All four steps run inside ONE `_with_locked_ledger` body via the new `ledger.transition_and_emit(...)` primitive (KTD-6). If steps 2 and 3 swap, A2 silent-exits before work units appear. If step 2 isn't in the same atomic snapshot, a crash between transition and emission leaves the ledger in an exited-looking-but-incomplete state — un-resumable cleanly. **The primitive is what makes this atomic** — without it (the round-1 framing's mistake), `record_verdict` and the emit would be separate locked writes with a torn-state window between them (feasibility F2 / adversarial F1).

**New ledger I-1 corollary** (documented in U11's contract update): "Phase-transition emission writes new pending units in the same atomic snapshot as the verdict-observation that triggers it; predicate recomputation runs on the post-emission unit set." Parallel to today's atomic predicate freshness invariant.

**Deliberate-fail test:** Edit `transition_and_emit` to recompute predicate BEFORE running the emitter on an A2 ledger fixture. Assert the predicate fires `met=true` after judge (incorrect). Confirm RED of the AE4 integration test. Restore.

### Serialized N parallel plan-loops (resolves G4)

Plan-loop units in N>1 mode (A2's 3 plan units) execute SERIALIZED — one `adapter.next_plan_step` call per tick, round-robin across ready plan units. The `next_plan_step(ledger)` signature is preserved unchanged. The top-level `plan_step` reflects the actively-advancing unit's value during its tick; other plan units are `dispatched`-but-idle.

**Round-robin tiebreaker (resolves feasibility F5 / adversarial F9):** the discriminator is the new per-unit `last_advanced_at` field. `orchestrator.pick_next_plan_unit_to_advance(ledger)` returns the ready plan unit (state `dispatched`, phase `plan`) with the oldest `last_advanced_at` (null sorts oldest, breaking initial ties by `units[]` declaration order). State lives in the ledger, so resume continues round-robin correctly.

**Stalled plan unit (resolves adversarial F3):** if a plan unit goes `stalled`, the round-robin ready-set excludes it (state ≠ `dispatched`). The predicate's plan-phase aggregation excludes stalled units from the "all plan units satisfy" universe (analogous to `terminal-skip`), AND the judge's `depends_on` gate treats a stalled dependency as a hard block surfaced to the operator via `auto-status` — NOT a silent hang. U6 specifies the recovery path.

### Start phase propagation

- `init_ledger` sets `loop_phase=recipe["phase_order"][0]`. For all V1 recipes (A1/A2/A4) this is `"plan"` (unchanged from v0.1.x). A3's `"work_sketch"` start is deferred with A3, so V1 never starts in a non-plan phase — but the helper is written to support it (the engine MUST NOT hardcode "plan is first") so v0.2.1's A3 needs no engine change.
- `phase_grammar.current_phase(ledger)` reads `ledger["loop_phase"]`, NEVER `phase_order[0]` directly. Load-bearing for resume.

    1. The round-robin ready-set EXCLUDES it (`pick_next_plan_unit_to_advance` only considers `state == "dispatched"` plan units), so the rotation continues across the remaining live plan units.
    2. The predicate's plan-phase aggregation EXCLUDES stalled units from the "all plan units satisfy" universe (analogous to `terminal-skip`) — a stalled unit does NOT block plan-met for the units that can still complete.
    3. The judge's (or any work unit's) `depends_on` gate treats a stalled dependency as a HARD BLOCK, surfaced to the operator via `auto-status` — NOT a silent hang. `auto-status` must report "judge blocked on stalled plan unit <id>" so the operator can intervene (re-dispatch or skip).
  - `recompute_predicate`'s plan-phase branch (single phase-aware `met` field per KTD-8; the two-field `phase_met` carving is DEFERRED to v0.2.1 with A3):
    - If ledger has top-level `gaps_open != None` AND no per-unit gaps_open fields populated, use top-level (A1 fast path).
    - Else aggregate over plan-phase units' per-unit `gaps_open`, EXCLUDING stalled units (per the stalled-recovery path above): plan-`met` requires all non-stalled plan units have per-unit `plan_step == "review_plan" AND gaps_open == 0`.
  - **Resume note (HLTD start-phase-propagation invariant):** if a run is paused mid-N-plan-loop advance, the ledger carries each plan unit's `plan_step`, `gaps_open`, and `last_advanced_at` (per-unit). ## System-Wide Impact

- **Interaction graph:** Adding recipes touches the engine's loop construction (`init_ledger`), tick loop (`_tick_body`, the former `_maybe_seam` now `_advance_phase_if_met`, `advance_plan_loop`), predicate evaluation (`recompute_predicate` — keeps a SINGLE phase-aware `met` field per KTD-8; the two-field `phase_met` carving is deferred to v0.2.1 with A3), the new `transition_and_emit` ledger primitive (KTD-6), the re-locked adapter contract's new `enumerate_plan_units` op (KTD-4), status reporting (`auto-status`), and resume (`auto-resume`). All except `auto-resume` read the new ledger fields directly. `auto-resume` re-reads the ledger fresh on suspend recovery, so it picks up recipe state automatically — no change needed.

- **Error propagation:** Recipe validation errors at engine load (`load_and_validate`) propagate to `lib/auto.py::run` which exits 1 with a clear message. Recipe validation errors at skill-write propagate to the skill which surfaces them to the user and refuses to write. The contract is `validate` (hard errors, used by both engine and skill) + `validate_and_lint` (additional editorial checks, skill only) per KTD-2. Same hard-error shape across both callers; lint warnings are skill-only and never reject a contract-valid recipe.

- **State lifecycle risks:** A v0.1.x ledger MUST be readable as a v0.2.0 ledger (additive-extension invariant). The R13 regression test enforces this. A run started under v0.2.0 with a workspace recipe survives the recipe file being deleted mid-run — `auto-resume` reads the ledger, not the recipe file, so topology info carries forward via the ledger's `recipe`/`phase_order`/`terminal_phase` fields (KTD-12). New atomic-ordering invariant (G3): the `transition_and_emit` primitive (KTD-6) writes new pending units in the same atomic snapshot as the verdict-observation that triggers it; because emit + recompute + write all happen in one `_with_locked_ledger` body, a crash between transition and emission is impossible by construction.

- **API surface parity:** `--recipe` is a new CLI flag. The `--adapter` flag is orthogonal and unchanged. The picker is a new UX surface on `commands/auto.md`. The authoring skill is a new skill. No existing command, flag, or behavior is removed or changed.

- **Integration coverage:** AE tests distributed across owning units (AE1+AE2 → U8, AE3 → U9, AE4 → U5b, plus the new A4 runtime end-to-end test → U5b). AE5 (A3) is deferred to v0.2.1. Each AE has a deliberate-fail check verified via the Edit tool. The three built-in recipes (A1/A2/A4) are the validator's conformance corpus. The R13 regression test (A1 ≡ v0.1.1) is the recipe-pipeline-wired-correctly smoke test. U10 is the verification gate, SPLIT per phase (each phase audits its own deliberate-fail checks per SG-2) so each phase lands as an audited, independently-meaningful release point.

- **Unchanged invariants:**
  - `dispatch_batch`, `ready_units`, `_is_ready`, `_dependency_satisfied`, `unit_is_terminal`, `record_verdict`, `transition`, `_atomic_write`, `_with_locked_ledger`, `advance_work_loop`, `_ready_fix_unit`, `_ready_reenqueue_unit`, `detect_and_halt_stalled` — all topology-agnostic, all unchanged.
  - The adapter contract (`docs/contracts/adapter-contract.md`) RE-LOCKS in V1 (KTD-4): the existing op signatures (incl. `next_plan_step`) are preserved, but the contract ADDS one new op `enumerate_plan_units(ledger) → list[unit_dict]` (the producer the emitters read), implemented by both `ce` and `native`. ## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| **Plan documents R11/R12 behavior the code doesn't fully wire** (dominant build-bug class from v0.1.1) | Centralized `lib/phase-grammar.py` helper as single decision point; AST lint test enforces no literal `loop_phase` reads outside the helper (KTD-3 — mechanical, not convention); three deliberate-fail tests in U5 each break a different drift mode. |
| **G1 — Dynamic unit emission silently broken** (A2/A4 ship as validating JSON but units never appear at runtime) | Explicit 3-emitter named-function registry in `lib/emitters.py` (U5b, KTD-5); the producer is the new `enumerate_plan_units` adapter op (KTD-4) so the emitters have source data; each emitter has unit tests + at least one edge case; AE4 + the new A4 runtime test assert the dynamic-emission path produces work units; atomic-ordering deliberate-fail confirms emit-then-recompute is correct. |
| **G2 — Predicate semantic conflation** (a non-default-phase recipe could silent-exit because `met` is overloaded for both phase-advance and run-exit) | The two-field carving (`met` + `phase_met`) is DEFERRED to v0.2.1 with A3 (the only recipe that needed it — KTD-8/KTD-14). In V1, `met` stays SINGLE-field but is phase-aware via the phase-grammar helper; since all V1 recipes (A1/A2/A4) use the default `phase_order`/`terminal_phase`, the conflation cannot occur (you can only exit from the terminal phase, which for all V1 recipes is `"work"`). v0.2.1 re-introduces the carving alongside A3's non-default phases. |
| **G3 — Predicate fires before dynamic units emit** (A2 judge terminal → ledger has zero work units → `all_units_terminal==true` → silent exit) | The new `ledger.transition_and_emit` primitive (KTD-6) + I-1 corollary: phase-transition emission writes new pending units in the same atomic snapshot as the triggering verdict-observation. Order inside the ONE locked body: (1) apply transition, (2) run (pure) emitter, (3) append units, (4) recompute predicate, (5) atomic write. Deliberate-fail test reverses recompute and emit, asserts AE4 RED. |
| **G4 — Scalar mirror under concurrent N>1 plan advance is structurally broken** | Resolved by KTD-4: plan-loop units in N>1 mode execute SERIALIZED (round-robin one-at-a-time per tick, tiebroken by per-unit `last_advanced_at`). The `next_plan_step` signature is preserved; the contract re-locks only to ADD `enumerate_plan_units`. Concurrent advance (a `unit_id` parameter on `next_plan_step`) deferred to v0.3.0. |
| **Re-entrancy deadlock if an emitter mutates the ledger** (KTD-6: the seam-handler already holds the lock; `_flock_run` opens a new fd per call) | Emitters are PURE functions (signature `emit(ledger, triggering_unit) → list[new_unit_dict]`) — they MUST NOT call `record_verdict`/`transition`/`set_loop`. The constraint is documented as `lib/emitters.py` module docstrings + the `transition_and_emit` docstring; a unit test asserts an emitter that calls a mutator deadlocks/errors (load-bearing, not convention). |
| **AskUserQuestion preview field doesn't render multi-line ASCII** | V1 fallback is short labels + descriptions; multi-line ASCII is an upgrade attempted in implementation; if it doesn't render, ship the fallback. Both paths tested in U8. |
| **Three independent ASCII renderers drift** (picker, skill, status) | Single renderer at `lib/topology-render.py` (KTD-10, U7); called from all three surfaces. Visual-vocabulary doc is the renderer's contract, not a separate spec. |
| **Authoring skill and engine loader validate differently** | `validate` is the shared contract (called by both); `validate_and_lint` is the skill-side editorial layer that adds non-blocking warnings (KTD-2). Test asserts skill output that passes `validate` is engine-acceptable. |
| **A1 recipe diverges from v0.1.1 behavior** | R13 regression test ("first-tick ledger from `--recipe a1` ≡ first-tick ledger from v0.1.1 invocation, modulo additive fields") is the wiring-correctness smoke test. Run on every PR. Per KTD-1, A1's canonical source is the `A1_BUILTIN` Python constant (NOT a removed legacy path) — a unit test asserts `A1_BUILTIN` equals the resolved `recipes/a1.json` so the two sources can't drift, and the constant is the runtime fallback so built-in install corruption can't break bare `/auto`. |
| **Round-1 fixes only the cited site (narrowing class)** | AST lint test (KTD-3) enforces structural single-source-of-truth mechanically; review-loop budget allocated for ≥3 rounds per institutional learning. |
| **Backward-compat breaks on a v0.1.x ledger** | U1 test loads a pinned v0.1.x fixture and asserts predicate evaluation matches v0.1.1's computation. Fail-stop if it diverges. |
